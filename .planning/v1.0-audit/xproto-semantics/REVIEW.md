> **Dedup note.** Sub-areas A and B independently found the *same* root defect — the disjoint byte-range lock maps (`lm.locks` vs `lm.unifiedLocks`). It is counted **once** as **H1** in the total. The raw per-sub-area tallies (A=2 HIGH, B=3 HIGH) sum to 5; after merging the duplicate cross-conflict finding the area carries **4 distinct HIGH**.

**Architecture invariants: hold.** All three sub-audits confirm the CLAUDE.md rules are respected at the boundary: protocol handlers stay protocol-only (XDR/SMB wire encode + dispatch); business logic, conflict decisions, and identity mapping live in `pkg/metadata` and `pkg/auth`; file handles remain opaque (owner IDs are compared by `==`, parsed only for NLM holder reporting/logging, never for conflict equality); WRITE follows the documented metadata→blockstore→attrs order. **Every finding here is cross-protocol *semantics* drift — a documented guarantee that the code does not actually deliver — not an invariant break.**

**Verdict: NEEDS-FIX before tag.** The headline is integrity-grade: the "unified cross-protocol lock manager" is **partially fictional at the byte-range layer**. SMB byte-range locks live in `lm.locks`; NLM + NFSv4 byte-range locks live in `lm.unifiedLocks`; neither the acquisition path nor the mandatory-lock I/O-enforcement path consults the other map. This directly contradicts the documented contract (`pkg/metadata/lock/types.go:84,143-147`: *"An NLM lock on bytes 0-100 will conflict with an SMB lock request for the same range"*) and on a share exported over both protocols allows silent cross-protocol data corruption. Two further HIGH faults sit in the *persistence* half (NLM/NFSv4 locks persist with `ShareName=""` and are silently dropped on restart; the memory LockStore's hand-written `cloneLock()` drops delegation + V2-lease identity fields). The ACL/attr boundary (sub-area C) is the healthiest — **zero HIGH** — but the SMB↔NFS translation is lossy in audit semantics, root identity, and DOS-attribute durability in ways the tests do not catch.

**Theme**: the *backbone* is sound (single internal NFSv4 ACL model, SMB translating at the wire boundary, opaque handle keying consistent across all three protocols, NFSv4 ACL round-trip essentially lossless, wire decoding bounded). The holes cluster in (1) the **byte-range lock map split** (acquisition + I/O enforcement) and (2) **cross-protocol persistence/identity loss** (ShareName, cloneLock, root SID, DOS bits, mode coherence). Both are exactly what same-protocol "tests pass" misses — each protocol's tests exercise only its own map and its own direction of translation.

## 2. HIGH findings (triaged, ranked by blast radius)

### Byte-range lock namespace split (the fault line)

- **H1 — SMB and NFS/NLM byte-range locks live in disjoint maps; cross-protocol conflict detection (LOCK-04) is not implemented despite being documented.** `pkg/metadata/lock/manager.go:669` (`lockInner` reads only `lm.locks`) vs `:1531` (`addUnifiedLockInner` reads only `lm.unifiedLocks`). SMB byte-range LOCK enters via `internal/adapter/smb/v2/handlers/lock.go:477` → `MetadataService.LockFile` → `service.go:492 lm.Lock` → `lockInner`, which conflict-checks **only** `lm.locks[handleKey]` via `IsLockConflicting`. NLM (`pkg/adapter/nfs/nlm.go:94`) and NFSv4 (`internal/adapter/nfs/v4/state/manager.go:1713`) enter via `AddUnifiedLock` → `addUnifiedLockInner`, which conflict-checks **only** `lm.unifiedLocks[handleKey]` via `ConflictsWith`. The two acquisition paths are provably disjoint and never cross-consult. *Why*: `types.go:84` (LOCK-04) and `:143-147` explicitly promise one shared namespace. A share exported over both SMB and NFS resolves to the **same** `Manager` for the same handle (`lockManagerForHandle` is keyed by handle, not protocol), so an NLM/NFSv4 exclusive lock on `[0,100)` is invisible to an SMB exclusive LOCK on the same range and vice versa — both grant, both clients believe they hold exclusivity → silent data corruption. The code's own comments confirm the split is deliberate-but-incomplete (`manager.go:1080`: byte-range ops *"consult lm.locks, not lm.unifiedLocks"*; `cross_protocol.go:149-152`: SMB byte-range locks *"are stored only in lm.locks today … not yet pushed through lockStore.PutLock (architectural debt)"*). The only existing bridge, `hasByteRangeLockConflictForLease` (`cross_protocol.go:156-187`), consults **both** views but is wired **only** into the lease-grant decision, not byte-range acquisition. *Fix*: make the byte-range acquisition path protocol-agnostic — have `lockInner` also scan `lm.unifiedLocks` (via `IsUnifiedLockConflicting`) and `addUnifiedLockInner` also scan `lm.locks` (via `IsLockConflicting`) before granting; or unify both paths onto one store keyed by `handleKey` with a protocol-opaque `OwnerID`. Add a `managertest` case: acquire an NLM exclusive lock, assert an overlapping SMB exclusive `Lock` fails, and the reverse.
  *Verifier note*: confirmed identically by both sub-audits A and B (counted once). Severity nuance from the adversarial pass: this is an **advisory** lock system, so the corruption window requires cooperating apps that honor advisory locks racing across SMB and NFS on the same file — narrower than mandatory locking, but a genuine violation of an explicit documented guarantee with **no** test catching it. HIGH retained.

- **H2 — Mandatory-lock I/O enforcement (`CheckForIO`/`CheckLockForIO`) ignores NLM/NFSv4 locks — an NFS lock never blocks an SMB write.** `pkg/metadata/lock/manager.go:888` (`CheckForIO` iterates **only** `lm.locks[handleKey]`). Its single caller is `MetadataService.CheckLockForIO` (`service.go:433`), whose callers are exactly six SMB handler sites: `write.go:321`, `read.go:304`, `set_info.go:1255`, `ioctl_sparse.go:454`, `ioctl_copychunk.go:432/442` (grep found no others). NLM/NFSv4 byte-range locks sit in `lm.unifiedLocks` and are therefore never seen by the SMB I/O conflict check. *Why*: SMB byte-range locks are **mandatory** (MS-FSA §2.1.4.10 — even the holder's shared lock blocks foreign writes). DittoFS enforces this only against same-protocol SMB locks: an NFS client holding an exclusive NLM/NFSv4 byte-range lock provides **zero** protection against a concurrent SMB write to the same range — the SMB write is allowed and corrupts the region the NFS client believes it has exclusively locked. The split is acknowledged at `manager.go:1080` but the I/O path never bridges it; the lease bridge in `cross_protocol.go` is wired only into lease-grant, not I/O enforcement. The `OpenID==""` "NFS/NLM POSIX" branch inside `lockInner` (`manager.go:684`) is dead for the current NLM path (NLM routes through `AddUnifiedLock`, not `Manager.Lock`), so it is not a bridge either. *Fix*: have `CheckForIO` also evaluate byte-range entries in `lm.unifiedLocks` (translate `UnifiedLock`→`FileLock` semantics or add a `UnifiedLock` I/O-conflict helper). Decide and **document** the cross-protocol mandatory-vs-advisory policy explicitly — the current silent gap is the worst outcome. (Couples with H1; same root, distinct enforcement path.)

### Cross-protocol persistence / recovery

- **H3 — NLM and NFSv4 locks persist with empty `ShareName`; the per-share recovery query silently drops them on restart.** `internal/adapter/nfs/v4/state/manager.go:1690-1694` (`ShareName:""`); NLM at `internal/adapter/nfs/nlm/handlers/lock.go:106-109` (ShareName left zero). Both reach `addUnifiedLockInner` → `persistUnifiedLockLocked` → `ToPersistedLock` (`pkg/metadata/lock/store.go:319`), which copies `pl.ShareName = lock.Owner.ShareName = ""` and — unlike the SMB/`FileLock` path `fileLockToPersisted` (`manager.go:1001`, which stamps `lm.shareName`) — **never** stamps the manager's share name. Recovery queries `LockQuery{ShareName: shareName}` (`service.go:135`) and `MatchesLock` returns false whenever `lk.ShareName != shareName` for a non-empty query (`store.go:202-204`). *Why*: every NLM and NFSv4 byte-range lock (and any NFSv4 delegation built the same way) is persisted with `ShareName=""` and therefore never matches the non-empty per-share recovery filter — silently never restored. NFS clients lose grace-period reclaim of their locks across a restart, while SMB locks (which **do** get `lm.shareName`) restore fine: an asymmetric, protocol-dependent lock loss. Manifests only across restart on a persistent LockStore backend (no corruption in steady state), but is silent and easy to miss. *Fix*: stamp `ShareName` in `ToPersistedLock` from the manager's `lm.shareName` the way `fileLockToPersisted` does, rather than trusting `lock.Owner.ShareName`; or have the NLM/NFSv4 adapters populate `LockOwner.ShareName`. Add a recovery test asserting an NLM/NFSv4 lock survives a per-share `RestoreLocks`.

- **H4 — Memory LockStore `cloneLock()` drops delegation ID + V2-lease identity fields on the persist round-trip.** `pkg/metadata/store/memory/locks.go:189-215`. `cloneLock` hand-copies a subset of `PersistedLock` and **omits** `DelegationID`, `DelegType`, `DelegBreaking`, `DelegRecalled`, `DelegRevoked`, `DelegNotificationMask`, `ParentLeaseKey`, `IsDirectory`, `IsTraditionalOplock`, `BreakingToRequired` — **and** `IsLegacyByteRange` (the restore-routing discriminator, see §3). It is invoked on the write path (`PutLock:52`, `putLockLocked:367`) and on **every** read (`GetLock:75`, `ListLocks:112`, `getLockLocked:380`, `listLocksLocked:399`), so the in-store copy is corrupted at `PutLock` time — no restart required. *Why*: (1) a persisted NFSv4 delegation loses `DelegationID` → `FromPersistedLock` (`store.go:428-442`) reconstructs the delegation branch **only** when `DelegationID != ""`, so it degrades to a plain byte-range/lease record — delegation identity destroyed; (2) a V2/dir lease loses `ParentLeaseKey`+`IsDirectory`+`IsTraditionalOplock`, breaking dir-lease parent-key break-suppression (`types.go:101-120`), the file-vs-dir distinction, and cross-tier oplock rules; (3) `BreakingToRequired` loss corrupts in-flight break-ACK validation. `ReclaimLease` in this same file (`locks.go:438-461`) calls `FromPersistedLock` on the already-corrupted stored copy, so the corruption manifests **within a single process lifetime**, not just across restart. Memory-store-specific: `badger`/`postgres` JSON-marshal the whole struct (`badger/locks.go:68/135/237`) and preserve all fields. **Uncaught**: `storetest/` has zero references to `DelegationID`/`ParentLeaseKey`/`IsTraditionalOplock`; `memory/locks_test.go` asserts only the `LeaseKey` round-trip. *Fix*: replace the hand-written `cloneLock` with a full struct copy (`clone := *lk`; then deep-copy the `[]byte` slices `LeaseKey`/`ParentLeaseKey`) so no field is silently dropped as `PersistedLock` evolves. Add a `storetest` case round-tripping a delegation and a V2 directory lease through `PutLock`/`GetLock` asserting `DelegationID`, `IsDirectory`, `ParentLeaseKey`, `IsTraditionalOplock`, `BreakingToRequired` (and `IsLegacyByteRange`) survive.

## 3. Triage downgrades / RESOLVED

Two HIGHs — both about `RestoreLocks` misrouting persisted NLM/NFSv4 byte-range locks into `lm.locks` — were **independently reported by sub-areas A and B and both refuted** by the adversarial pass against current `HEAD`.

- **A/B "RestoreLocks routes persisted NLM/NFSv4 byte-range locks into the wrong map (post-restart orphan leak + behavior flip)" → RESOLVED.** Both findings assert that `RestoreLocks` classifies records by *shape* — via a helper `isByteRangePersistedLock` checking "no 16-byte LeaseKey and no DelegationID" at `manager.go:1100-1103` — which cannot tell a plain SMB byte-range lock from an NLM/NFSv4 unified lock, so restored unified locks would land in `lm.locks`. **The bug does not exist in current code; it has already been fixed.** The actual code routes on an **explicit persisted tag**: `RestoreLocks` (`manager.go:1092`) branches on `pl.IsLegacyByteRange` — true → `lm.locks`, false → `lm.unifiedLocks` via `FromPersistedLock`. The cited `isByteRangePersistedLock` helper **no longer exists** anywhere in `pkg/metadata/lock/` (grep is empty; lines 1100-1103 are now the doc comment for `fileLockFromPersisted`). The tag is set true **only** in `fileLockToPersisted` (`manager.go:1008`, the legacy SMB path); the unified path `ToPersistedLock` (`store.go:307`) never sets it (defaults false), so NLM/NFSv4 locks correctly route back into `lm.unifiedLocks` — visible to `ListUnifiedLocks`/`AddUnifiedLock`/`RemoveUnifiedLock`. The doc comment on the `IsLegacyByteRange` field (`store.go:58-64`) explicitly documents it was added as the remediation for exactly this failure mode. The findings' load-bearing evidence — that `TestAddUnifiedLock_PersistsAndReloads` (`persist_test.go:51-87`) is *currently failing* with "should have 1 item(s), but has 0" — was **falsified**: the test PASSES (asserts the restored lock appears in `ListUnifiedLocks` with `OwnerID nlm:client-x`). Both findings were written against an older revision (the lock files are uncommitted-modified per `git status`); at the cited locations the described bug is resolved.
  *Caveat carried forward*: `IsLegacyByteRange` **is itself dropped by the memory store's `cloneLock`** (H4). So while the *routing logic* is correct, a legacy SMB byte-range record round-tripped through the **memory** LockStore would lose its discriminator and misroute on restore — but that is the memory-store `cloneLock` defect (H4), not a `RestoreLocks` defect. The routing fix stands; H4 is the live residue.

## 4. MED findings (defer as issues unless trivially co-fixable)

**A — length/range:**
- `ToPersistedLock` drops `IsZeroByte` — the unified persist path (`store.go:309`/`FromPersistedLock`) never sets/reads it, while the `FileLock` path (`manager.go:999-1011/1069-1077`) round-trips it. Latent: `Length==0` is overloaded (SMB zero-byte vs NFS to-EOF, `store.go:52-56`); any future zero-byte lock persisted through the unified path silently downgrades to a to-EOF lock, inverting conflict + I/O decisions. Make both persist paths symmetric.
- NLM/NFSv4 byte-range LOCK has **no `offset+length` uint64 overflow guard** while SMB has one (`smb/v2/handlers/lock.go:407` → `StatusInvalidLockRange`). NLM (`internal/adapter/nfs/nlm/handlers/lock.go:112`) passes wire offset/length straight to `AddUnifiedLock` → `rangeLast = offset+length-1` (`types.go:389-394`) wraps for `offset+length > 2^64`; the lock-package doc even states callers must reject this (`types.go:378,400`). A malicious `offset=2^64-1,length=2` collapses the range → under-conflict detection, divergent from the same range over SMB (rejected). Centralize the overflow check in `MetadataService.LockFile`/the manager so all protocols share one validation.

**C — ACL/SD & attr translation:**
- **NFS chmod silently wipes SMB-managed DOS attributes** (READONLY/SYSTEM/ARCHIVE/SPARSE/COMPRESSED). `pkg/metadata/file_modify.go:250` does `file.Mode = newMode` (full replacement); NFS SETATTR delivers a mode masked to `0o7777` (v3 `setattr.go:436`; v4 has no high bits), clearing every DOS high bit (`converters.go:32-64`). A file marked READONLY/SYSTEM via SMB loses the attribute the instant any NFS `chmod` touches its mode → Windows clients re-read the wrong attribute set. Fix: preserve the DOS high-bit nibble on replacement, e.g. `file.Mode = (file.Mode &^ 0o7777) | (newMode & 0o7777)`, mirroring the merge already done SMB-side at `set_info.go:527`.
- **NFSv4 AUDIT ACEs emitted into the SMB DACL instead of the SACL.** `internal/adapter/smb/v2/handlers/security.go:331` — `buildDACL` calls `nfsToWindowsACEType` which returns ok=true for `ACE4_SYSTEM_AUDIT_ACE_TYPE` (→ `systemAuditACEType 0x02`), so an audit ACE is written into the DACL body. Per MS-DTYP §2.4.5/§2.4.6 audit ACEs belong in the SACL; `buildEmptySACL` (`:375`) always emits a 0-ACE SACL. The DACL becomes malformed and the audit ACE is subjected to DACL access evaluation it should never participate in. Fix: skip AUDIT (and ALARM) ACEs in `buildDACL`; route AUDIT into a real `buildSACL`.
- **SMB SACL section is write-only stubbed — audit ACEs from SET_INFO Security are silently discarded.** `security.go:428` skips `offsetSACL` (`r.Skip(4) // SACL parsing not implemented`); the build side only ever writes an empty SACL. A client that submits SACL audit ACEs gets accept-and-discard with a success status, and a subsequent QUERY returns an empty SACL. Fix: parse the SACL into AUDIT/ALARM ACEs (reuse `parseDACL`), store them (the NFSv4 model already holds AUDIT type), serialize them back; or return `STATUS_INVALID_PARAMETER`/`NOT_SUPPORTED` rather than silently dropping.
- **Audit-ACE success/failure flags dropped by both flag translators.** `pkg/metadata/acl/flags.go:16` — `NFSv4FlagsToWindowsFlags`/`WindowsFlagsToNFSv4Flags` translate only the five inheritance bits, never `ACE4_SUCCESSFUL_ACCESS_ACE_FLAG (0x10)`/`ACE4_FAILED_ACCESS_ACE_FLAG (0x20)` ↔ Windows `SUCCESSFUL/FAILED_ACCESS_ACE_FLAG (0x40/0x80)`. An audit ACE that survived type translation would still lose its success/failure selector (a dead audit ACE). Independent latent bug; surfaces as soon as the SACL/AUDIT gaps are addressed. Add the 0x10↔0x40 / 0x20↔0x80 mapping to both functions.
- **Root (uid 0) and SYSTEM@/ADMINISTRATORS@ SID mapping is asymmetric — identity lost on round-trip.** `pkg/auth/sid/mapper.go:84` — `UserSID(0)` → `WellKnownAdministrators S-1-5-32-544` and `principalToSID` maps `SYSTEM@`→`S-1-5-18`, `ADMINISTRATORS@`→`S-1-5-32-544` (`security.go:294-297`), but the reverse is missing: `UIDFromSID`/`GIDFromSID` only recognize `S-1-5-21` domain SIDs, and `SIDToPrincipal` (`mapper.go:235-260`) has no case for `S-1-5-18`/`S-1-5-32-544`, so they degrade to literal `sid:S-1-5-…`. Net: (a) a root-owned file's OWNER@ ACE round-trips back as `sid:S-1-5-32-544`, not OWNER@; (b) a SET_INFO Security with owner=`S-1-5-32-544` silently no-ops the owner change (`ParseSecurityDescriptor` leaves `ownerUID` nil, `security.go:435-442`); (c) synthesized SYSTEM@/ADMINISTRATORS@ ACEs never map back. Loss of canonical identity for the most security-sensitive principal. Fix: make `SIDToPrincipal` recognize `S-1-5-18`→SYSTEM@ and `S-1-5-32-544`→ADMINISTRATORS@ (and `UIDFromSID(S-1-5-32-544)`→uid 0) so `principalToSID`/`SIDToPrincipal` are inverses.

## 5. LOW findings

**A — length/range:**
- Byte-range unlock matches only on `(owner,offset,length)`; `IsZeroByte` is not in the match key (`manager.go:742` `unlockInner`; `RemoveUnifiedLock` via `Overlaps`). If an owner ever held both a `Length==0` zero-byte lock and a `Length==0` to-EOF lock at the same offset, the unlock keyed on `(owner,0,0)` removes whichever is first in slice order. Latent today (disjoint maps) but goes **live the moment the maps are unified to fix H1/H2** — include `IsZeroByte`/an explicit lock-kind in the unlock predicate.

**B — handle/id:**
- `ToPersistedLock` invariant violation (both `Lease` and `Delegation` set) is **logged but silently mispersisted** — `store.go:299-352` logs Error then persists only the lease branch (`&& lock.Lease == nil` guard at `:341`), writing a partial/ambiguous record rather than rejecting. Return an error on this invariant violation, or separate Lease/Delegation into distinct record kinds at the type level.

**C — ACL/attr:**
- **ALARM ACEs dropped both directions while `FATTR4_ACLSUPPORT` advertises ALARM.** `security.go:60` — `nfsToWindowsACEType`/`windowsToNFSACEType` have no case for `ACE4_SYSTEM_ALARM_ACE_TYPE (0x03)`; `buildDACL`/`parseDACL` `continue` past it. Meanwhile `EncodeACLSupportAttr` (`attrs/acl.go:117-119`) returns `FullACLSupport` including `ACL4_SUPPORT_ALARM_ACL` and `ValidateACE` accepts ALARM — so the server advertises a capability it cannot round-trip. Spec-honesty/silent-loss (Windows has no SYSTEM_ALARM analog). Lowest-surface fix: drop `ACL4_SUPPORT_ALARM_ACL` from `FullACLSupport`, advertising only ALLOW/DENY/AUDIT.
- **Setting an ACL never recomputes the POSIX mode; `DeriveMode` is dead production code.** `file_modify.go:387` stores `file.ACL = attrs.ACL` without calling `acl.DeriveMode` (the inverse coupling exists — chmod calls `AdjustACLForMode` at `:255`). `DeriveMode` (`acl/mode.go:16`) has zero non-test callers. An SMB DACL tighten leaves NFS GETATTR `FATTR4_MODE` stale (RFC 7530 §6.4.1 expects mode/ACL coherence). Pick one: call `DeriveMode` on set-ACL, or delete the dead function.
- **Same file reports different fileid/inode across NFS and SMB.** NFS `FATTR4_FILEID`/`MOUNTED_ON_FILEID` = first 8 bytes of `SHA-256(handle)` (`encode.go:557`, `types.go:92-98`); SMB `FileInternalInformation`/`FileIdInformation` derive from the metadata file UUID `file.ID` (`query_info.go:903`, `handler.go:1594-1602`). Breaks cross-protocol hardlink detection and identity caching; `FATTR4_FSID` is also hard-coded `(0,1)` vs SMB volume serial `0x12345678`. Derive both from one stable source, or document the divergence as a known v1.0 limitation.
- **NFSv3 timestamps truncate to uint32 seconds (year-2106 wrap) vs SMB FILETIME full range.** `internal/adapter/nfs/xdr/attributes.go:95` encodes `uint32(Atime.Unix())` (correct per RFC 1813, but a far-future SMB-set FILETIME reads back wrong over NFSv3). NFSv4 uses uint64 seconds and is unaffected. Clamp/saturate to `0xFFFFFFFF` rather than wrapping, and document the NFSv3 ceiling.

## 6. Verified-correct (checked, no finding)

**Lock layer (A+B):**
- `handleKey` is consistently `string(metadata.FileHandle)` across SMB (`service.go:482`), NLM (`nlm/handlers/lock.go:165`, `pkg/adapter/nfs/nlm.go:93`), and NFSv4 (`v4/state/manager.go:1712`) — all three protocols target the same per-file bucket; the file-handle dimension itself is consistent.
- SMB `FileLock.OpenID = hex(FileID)` and NFS/NLM fallback owner `"session:N"` do **not** collide as strings (hex vs `session:` prefix); within a single map the per-open vs per-session ownership comparison is internally consistent.
- The `lock.PersistedLock` **conversion layer itself is correct** — `LeaseKey`/`ParentLeaseKey` 16-byte round-trip (`store.go:325-337/383-399`) and `BreakingToRequired` backfill (`store.go:411-417`) are sound. The loss in H4 is purely in the memory store's `cloneLock`, not in `ToPersistedLock`/`FromPersistedLock`.
- FileHandle opacity preserved at the handler boundary: handlers pass raw FH/`MetadataHandle` into `LockFile` and never parse it; lock owner IDs are opaque to the Manager (compared by `==`; `extractClientID`/`parseNLMOwnerID` at `cross_protocol.go:286-323` parse only for NLM4_DENIED holder reporting + INFO logging, **not** conflict-equality decisions).
- `RangesOverlap`/`rangeLast` inclusive-end arithmetic (`types.go:381-394`) correctly handles unbounded (`length=0`→EOF) and avoids the `2^64-1` single-byte wrap; `IsZeroByte` is persisted on the `FileLock` path so SMB zero-byte locks are not confused with NFS to-EOF on restart.

**ACL/attr layer (C):**
- `FILETIME`↔`time.Time` conversion (`types/filetime.go:21-43`) is correct: tick/epoch arithmetic split to avoid int64 overflow beyond 2262, with `>MaxInt64`/zero guards. Round-trips cleanly.
- NFSv4 **inheritance**-flag translation (`acl/flags.go`) correctly maps NFSv4 `0x80 INHERITED_ACE`→Windows `0x10` (avoids the naive-0xFF-truncation trap) and all five inheritance bits bidirectionally.
- `FATTR4_MODE` masks `Mode&0o7777` (`encode.go:569`) and NFSv3 `MetadataToNFS` masks `Mode&0o7777` (`attributes.go:82`), so SMB DOS high mode-bits do **not** leak into the NFS-reported mode.
- `GENERIC_*` expansion (`acl/generic.go ExpandGenericMask`) is applied at the SD-parse boundary (`security.go:566`) per MS-DTYP §2.5.3 with generic bits stripped post-expansion; the file-object `GenericMapping` table matches `0x00120089/0x00120116/0x001200A0/0x001F01FF`.
- SID binary encode/decode (`sid/sid.go`) handles 6-byte big-endian IdentifierAuthority + little-endian sub-authorities with length guards (`DecodeSID:60-83`) and bounds checks in the `parseDACL` ACE walk (`security.go:526-541`).
- `DecodeACLAttr` (`attrs/acl.go:68`) and `parseDACL` (`security.go:518`) both bound ACE count by `acl.MaxACECount (128)` before allocating — no unbounded-allocation DoS from a malicious ACE count.
- `SIDMapper` RID scheme (`user=uid*2+1000`, `group=gid*2+1001`) guarantees `UserSID(n)!=GroupSID(n)`; `UIDFromSID`/`GIDFromSID` distinguish via even/odd parity for domain SIDs.
- `ValidateACL` intentionally does **not** enforce canonical ACE ordering — correct per MS-DTYP §2.4.5 (ACL is an unordered array; canonical order is a Windows-editor presentation convention); evaluation walks stored order deterministically.
- `AdjustACLForMode` (`acl/mode.go:59`) preserves non-special (explicit user/group) ACEs unchanged on chmod and only rewrites OWNER@/GROUP@/EVERYONE@ rwx bits, preserving non-rwx admin bits — matches RFC 7530 §6.4.1.
