**Architecture invariants: clean.** All five sub-audits confirm the CLAUDE.md rules hold for this area:
- **Invariant 1** (business logic in `pkg/metadata`, not handlers) — permission/squash checks live in the service layer; name-correctness has one defense-in-depth gap (L: `/`/NUL not centrally rejected, §5).
- **Invariant 2** (`*metadata.AuthContext` threaded) — every mutating op carries it and runs `checkWritePermission`/`CheckParentCreateAccess`/`checkDeletePermission`/sticky-bit.
- **Invariant 3** (opaque handles, decode-for-routing-only) — the store stamps the share name via `EncodeShareHandle`; the service decodes the same field; routing is internally consistent. One footgun: a `:` in a share name produces undecodable handles (MED, §4).
- **Invariant 5** (WRITE = Prepare → BlockStore → Commit) — the two-phase `PrepareWrite`/`CommitWrite` pattern satisfies it cleanly.
- **Invariant 6** (return `metadata.ExportError`; log expected at Debug) — **one violation**: malformed/stale handles return plain `fmt.Errorf` → NFS3ERR_IO/SERVERFAULT instead of BADHANDLE/STALE, logged at Error (MED, §4).

**Verdict: PATCH-grade.** No correctness backbone is broken; the postgres store is in solid shape post-#838/#839/#853 (SQL fully parameterized, NULL/ON-CONFLICT semantics correct, tx-bound mutations honor rollback, idempotent migrations). The single HIGH is a **test-coverage hole, not a live defect** — a meaningful subset of the `MetadataStore` interface (`DeleteShare`, `GetUsedBytes`, `GetFileByPayloadID`, filesystem meta/stats/caps, server config, `Healthcheck`) has **zero** cross-backend conformance coverage, so any backend could diverge or regress on those surfaces green. This is the same class that hid the three postgres bugs (which were entirely CI-skipped pre-#853). The recurring theme across the area is **cross-backend divergence the suite cannot catch**: an incomplete deep-copy (memory aliases `FileAttr.ACL`), badger conformance gated behind `//go:build integration` (dark by default), badger value-log GC never invoked, and several documented contract clauses (pagination, atomic rollback, concurrent-CRUD safety) asserted nowhere. The MED set is dominated by **swallowed-error / lifecycle-leak / TOCTOU** items — exactly what "tests pass" misses.

## 2. HIGH findings (triaged, ranked by blast radius)

### Conformance coverage

- **H1 — Whole `MetadataStore` interface methods have ZERO cross-backend conformance coverage.** `pkg/metadata/store.go:108` (`GetFileByPayloadID`), `:119`/`:123` (`Get`/`PutFilesystemMeta`), `:164` (`DeleteShare`, Shares block), `:192`/`:195` (`Set`/`GetServerConfig`), `:202`/`:208` (`GetFilesystemCapabilities`/`Statistics`), `:377` (`GetUsedBytes`), `:395` (`Healthcheck`). A grep across the entire `pkg/metadata/storetest/` tree returns **zero** matches for all eight method names (grep validity confirmed — `CreateShare`/`PutFile`/`GetFile` do match in the same dir). `RunConformanceSuite` (`suite.go:43`) runs only FileOps/DirOps/Permissions/DurableHandles/FileBlockOps/BlockRefOps/TruncateBlockRefOps/ObjectIDOps/INV02Fuzz — none touch these surfaces; `backup_conformance.go`/`reset_conformance.go` reference none either. `GetUsedBytes` is exercised only in `store/memory/counter_test.go` (memory-only, not the shared suite); `DeleteShare` and `GetFileByPayloadID` have **no** backend test at all.
  **Why it matters:** all three surfaces are production-consumed — `DeleteShare` from `internal/controlplane/api/handlers/shares.go`, `GetFileByPayloadID` from `pkg/controlplane/runtime/shares/coordinator.go` (flusher), `GetUsedBytes` from `pkg/controlplane/runtime/runtime.go` (statfs/quota). `GetUsedBytes` is maintained as an incremental atomic counter per backend (`store.go:376` "O(1) read from an atomic counter"), so it is highly prone to drift — does it decrement on `DeleteFile`? on truncate-down `SetAttr`? count only regular files? rebuild after `Restore`? **None verified on any backend.** A backend can panic, leak, or return wrong values and still pass the full suite green. This is the exact CI-blind class that hid the three #853 postgres bugs.
  **Verifier rationale:** confirmed by reading the code — all eight methods exist at the cited lines, the grep is clean, and all three backends implement each, so cross-backend drift is a real untested risk. Inaccuracies in the original finding were purely cosmetic (a `store/` path-prefix omission; `DeleteShare` cited as `interface.go:161` but the suite-exercised declaration is `store.go:164`). The substantive claim is exactly true.
  **Fix:** add shared-suite scenarios — `DeleteShare`-then-`ListShares`-excludes-it + `GetRootHandle` NotFound; `GetUsedBytes` reflects a known total after create/write/truncate/delete identically on all three backends; `GetFileByPayloadID` round-trip + miss; filesystem meta/stats/caps round-trip; server-config round-trip; `Healthcheck` returns StatusHealthy. Run across memory/badger/postgres.

## 3. Triage downgrades / RESOLVED during this pass

- **C-HIGH "ListChildren cursor/limit pagination documented but never exercised" → MED.** Substantive claim holds (the contract lives at `store.go:65-69`, not the cited `interface.go:65` which is `Move`): the only test, `testListDirectory` (`storetest/dir_ops.go:63-103`), calls `ListChildren(ctx, root, "", 100)` against 3 entries and asserts `nextCursor==""` — single-page, oversized-limit only. No conformance test passes a small limit to force >1 page, threads `nextCursor` back across a real boundary asserting no dup/missing entries, or asserts `limit==0` selects a default. The near-mitigation `walkShareFiles` (`inv02_fuzz.go:550-579`) threads `cursor=next` but with `limit=0` and (per its own comment) creates files only at the share root, so the default page size returns everything in page 1 — the boundary is never forced. Pagination is backend-divergent (postgres keyset vs badger iterator prefix vs memory map), so a wrong `nextCursor` / dropped-or-doubled boundary entry breaks large-dir READDIR while passing the suite. **Downgraded to MED** because it is a missing-test finding, not a demonstrated live defect, and the original cite was the wrong file.

- **C-HIGH (framed LOW) "CreateShare ErrAlreadyExists clause never asserted" → LOW (test-hygiene).** **Refuted on the substantive claim.** Cited `interface.go:152-155` is the Quota Management header, not CreateShare docs; the clause lives at `store.go:154`. The core claim "duplicate-share path is unguarded across all three backends" is **false** — all three actively guard it and return `StoreError{Code: ErrAlreadyExists}`: memory `store/memory/transaction.go:471-490`, badger `store/badger/shares.go:118-129`, postgres `store/postgres/shares.go:126-141`. The data-loss/share-hijack premise cannot occur because every backend rejects the duplicate. Postgres additionally has a dedicated `store/postgres/createshare_contract_test.go:61-65` asserting `errors.As(...) && Code==ErrAlreadyExists`. **Residual:** the cross-backend `storetest` suite lacks a duplicate-CreateShare scenario, so memory/badger get no *shared-suite* assertion — a minor LOW coverage gap (now listed in §5), not a HIGH bug.

- **C-HIGH "GenerateHandle stability-across-restart and determinism asserted nowhere" → RESOLVED.** **Refuted.** The finding (1) misreads the contract — `GenerateHandle` is *intentionally* non-deterministic (both backends mint a fresh `uuid.New()` per call: memory `store.go:537`, postgres `shares.go:25`); "must be stable" (`store.go:99-102`) means the minted handle, once persisted as the file's identity, stays resolvable across restart, NOT that re-calling reproduces the same bytes for a `(share,path)`. Asserting determinism-per-path would assert a property the design rejects. (2) Targets the wrong test — the cited `objectid_lookup.go:99-105` comment concerns **ObjectID** (the content-addressed dedup id), not file handles. (3) Is factually wrong that no close+reopen handle re-resolution test exists: `TestBadgerStore_PermissionPersistence` (`store/badger/permission_integration_test.go:82-139`) creates a badger store at a real dbPath, captures `rootHandle`, `Close()`s, reopens against the **same** path, and re-resolves the identical handle via `GetFile` successfully — exactly the cached-handle-survives-restart property. Postgres adds `TestPostgres_StoreID_SurvivesResetAndReopen` (`store_id_test.go`) + a reopen conformance factory (`postgres_conformance_test.go:64-92`). The ESTALE risk is guarded.

## 4. MED findings (defer as issues unless trivially co-fixable)

**postgres (2):**
- **`DeleteShare` orphans all file rows** (`store/postgres/shares.go:205-224`, also `transaction.go:858-877`) — runs only `DELETE FROM shares WHERE share_name=$1`; `shares` is referenced *by* `files` (no cascade `shares→files`), so every `files`/`parent_child_map`/`link_counts`/`file_block_refs` row is orphaned. Violates the `store.go:161` "removes a share and all its metadata" contract (memory does delete the files; postgres does not). Worse, the orphaned root inode at path `/` keeps the partial unique index `unique_share_path_hash_active` occupied → same-name recreation fails with `ErrAlreadyExists` for a share the operator believes deleted. **Held at MED** because the control-plane `RemoveShare` path does not currently invoke the metadata `DeleteShare` (reachable via REST/conformance) and badger shares the gap. Fix: delete the share's `files` first inside the tx (descendants cascade), then the `shares` row; mirror memory; apply to pool- and tx-paths.
- **`usedBytes` counter double-counts on tx retry** (`store/postgres/transaction.go:276,306,375` mutation vs `:84-114` retry loop) — `PutFile`/`DeleteFile` call `usedBytes.Add(delta)` inside `fn()`; `WithTransaction` re-runs `fn` on 40P01/40001 up to 3× without reversing the in-memory delta → statfs byte counter drifts under contention (resets only on restart via `initUsedBytesCounter`). Not used for quota enforcement, so blast radius is reported free-space only. Fix: accumulate `pendingDelta` on the tx struct and apply `usedBytes.Add` once after a successful `Commit`.

**memory + badger (4):**
- **Memory shares `FileAttr.ACL` pointer (incomplete deep-copy)** (`store/memory/store.go:496-507`; also `files.go:299-301,360-361`, `transaction.go:81-84,308-309`) — `GetFile`/`ListChildren`/`PutFile` deep-copy only `Blocks` via `cloneBlocks` (T-12-09) and leave `FileAttr.ACL` (`*acl.ACL` holding `[]ACE`) shared by pointer between caller and stored row. Badger round-trips through JSON and never aliases → identical inputs, divergent aliasing. **Exact class** of the area-5 `cloneLock` field-drop bug; ACL is the only other reference-bearing field in `FileAttr`. An in-place ACE edit can silently corrupt stored permissions. Fix: add `cloneACL` and apply alongside `cloneBlocks` at all five sites; add a PutFile-then-mutate-caller-ACL conformance scenario (both directions).
- **Badger conformance gated behind `//go:build integration`** (`store/badger/badger_conformance_test.go:1`) — `TestConformance`/`TestBackupConformance`/`TestResetThenRestoreConformance`/`TestLockPersistenceConformance` never run under default `go test ./...` (memory equivalent has no tag). Same CI-visibility shape that hid the postgres bugs; any badger-specific contract regression passes green. Fix: confirm CI runs `-tags=integration` on `store/badger/...`, or drop the tag from the pure in-process tests (they use `t.TempDir`, no external service).
- **Badger value-log GC (`RunValueLogGC`) never invoked anywhere** (`store/badger/store.go` package-wide) — opens with `SyncWrites=true`; every refcount/state/rollup mutation is a full read-modify-write rewrite. BadgerDB's value log is reclaimed ONLY via explicit `RunValueLogGC`; without it a long-running persistent backend (docstring: "Production environments requiring persistence") grows the value log unbounded until disk fills, no symptom until then. Fix: background ticker goroutine running `RunValueLogGC(0.5)` until it returns non-nil, stopped cleanly in `Close()`.
- **Badger tx list/enumerate FileBlock methods open a fresh `db.View`** (`store/badger/objects.go:793-803`) — `tx.ListPending`/`tx.ListFileBlocks`/`tx.EnumerateFileBlocks` delegate to `tx.store.X`, each opening its own snapshot taken at call time, so a `tx.Put` then `tx.ListFileBlocks` in the same `WithTransaction` misses the uncommitted write. The Put/Get/AddRef/Inc/Dec/GetByHash methods were explicitly fixed to bind to `tx.txn` (`objects.go:543`); the list methods were left delegating. Read-after-write within a tx is a normal expectation; memory list helpers read live maps under the held lock → silent cross-backend divergence, no test covers it. Fix: reimplement the three on `badgerTransaction` to iterate the active `tx.txn`, or document the limitation and assert it.

**metadata service / routing (3):**
- **Malformed/stale handles return generic error → NFS3ERR_IO/SERVERFAULT instead of BADHANDLE/STALE** (`types.go:65,72,77`; `service.go:329,349`) — `DecodeFileHandle` and `storeForHandle`/`lockManagerForHandle`/`GetStoreForShare` return plain `fmt.Errorf`, so `common.lookupErrorRow`'s `errors.As(&StoreError)` fails and falls through to `defaultCodes` (`errmap.go:25-29`). A ready-made `NewInvalidHandleError()` (`errors/errors.go:228` → BADHANDLE/STATUS_INVALID_HANDLE) is unused on this path. **Violates invariant 6**: clients get a server-fault (so they don't drop/refresh the handle) and the event is logged at Error — a hostile client can spam Error logs with garbage handles. Fix: return `NewInvalidHandleError()` for malformed handles and an `ErrStaleHandle` StoreError for decoded-but-unknown shares.
- **Share name containing `:` accepted but produces undecodable handles** (`types.go:42-48,60-81`; `runtime/shares/service.go:332-334`) — handle format is `<shareName>:<uuid>` split on the FIRST `:`; `AddShare` validates only `Name != ""` and REST `normalizeShareName` doesn't reject `:`. A share named `/foo:bar` yields a handle whose UUID-parse of `bar:<uuid>` fails → every file inaccessible (and, per the finding above, as a server fault). Silent operator footgun undermining the "handles encode share identity" invariant. Fix: reject `:` in share names at `AddShare`/Create (ideally length-prefix the share component); add a colon-bearing-name validation test.
- **`RemoveShare` never deregisters store/lockManager/quota/notifier from `MetadataService`** (`service.go:39-48,130-203`; `runtime/shares/service.go:774-814`) — `MetadataService` has `RegisterStoreForShare` but no `Unregister`; `RemoveShare` deletes only its own registry. `stores`/`lockManagers`/`unifiedViews`/`dirChangeNotifiers`/`quotas` retain the entry forever → unbounded map growth across AddShare/RemoveShare churn (lock managers hold timers), stale routing (a removed-share handle still resolves to a live store), and same-name re-add silently reuses the stale lock manager (`RegisterStoreForShare` early-returns when `lockManagers[name]` exists, line 143-145). Fix: add `RemoveStoreForShare(name)` deleting all five maps under `s.mu` (abort grace timer / close lock manager) and call it from `RemoveShare`.

**file / dir / IO ops (3):**
- **`RemoveFile` link-count read is outside the transaction (TOCTOU with `CreateHardLink`)** (`file_remove.go:85,102-137`) — `linkCount` read via `store.GetLinkCount` at line 85 BEFORE `WithTransaction`; the tx branches on/writes the stale value (`linkCount-1` or 0), never re-reading via `tx.GetLinkCount`. A concurrent `CreateHardLink` (which atomically Get+Set inside its own tx, `file_create.go:138-141`) bumps the count in the window → `RemoveFile` sets `nlink=0` while a valid link still references the file → orphaned/last-link-removed, content becomes eligible for deletion. Because the count is never read inside the tx, badger's read-write conflict detection has no key to trip, so it will NOT auto-retry; postgres/memory `SetLinkCount` are blind writes. Fix: move the `GetLinkCount` read inside the closure (`tx.GetLinkCount`) so read+decision+write are atomic, matching `CreateHardLink`.
- **`Move` swallows transaction-critical errors and commits partial rename** (`file_modify.go:598,605,609,618,630,636,639`) — `tx.SetParent`, the dir link-count `SetLinkCount`s, `tx.PutFile(srcFile)` (new Path+ctime), `tx.PutFile(srcDir)`, `tx.PutFile(dstDir)` are all discarded with `_ =`; `updateDescendantPaths` errors are logged-and-ignored; the closure returns `nil` unconditionally → `WithTransaction` commits regardless. Only the child-mapping ops propagate errors. If `PutFile(srcFile)` fails the entry is relinked but the File row keeps the OLD Path and link counts stay wrong — stale `File.Path` breaks the Path-keyed postgres snapshot manifest/restore and descendant-path rewrite; a directory move can permanently skew parent nlink. Defeats the tx all-or-nothing guarantee. Fix: return the error from each tx call (at minimum `PutFile(srcFile)`, `SetParent`, link-count updates) so a failure rolls the whole rename back; propagate `updateDescendantPaths` or document why a partial commit is acceptable.
- **`CookieManager` reverse map grows unbounded for process lifetime (slow memory DoS)** (`cookies.go:49,70-83`) — `GenerateCookie` stores every `(cookie→name)` pair in the `cookieToToken` sync.Map and never removes individual entries; `Clear()`/`ClearForDirectory()` are the only eviction paths and neither is called from non-test code (`ClearForDirectory` is an empty no-op). Every READDIR/READDIRPLUS entry inserts one map entry for the MetadataService lifetime → steady growth / eventual OOM; no TTL/LRU/cap. The mtime-based verifier handles staleness for correctness but reclaims nothing. Fix: bound the reverse map (LRU cap or periodic Clear), or encode the resume token in the cookie so no reverse map is needed; at minimum remove the dead no-op `ClearForDirectory` or wire real eviction on `notifyDirChange`.

**store-contract conformance (3):**
- **ListChildren pagination semantics never exercised** (`store.go:65-69`) — downgraded from HIGH; see §3. Add a multi-page scenario (N>limit children, page with small K threading `nextCursor` until empty, assert union == full set with no dupes) + a `limit==0` default-behavior assertion.
- **Suite codifies memory store's broken tx rollback as conformant** (`storetest/file_block_ops.go:1033-1090`, esp. `:1042`) — `testTx_IncrementRefCount_RollsBack` string-matches `"*memory.MemoryMetadataStore"` and RELAXES the assertion: memory leaves increments stuck on rollback (best-effort txn) and the test tolerates it. The `WithTransaction` contract (`interface.go:257-269`) is unconditional (error → roll back). This is the **suite-asserts-a-bug-as-correct** anti-pattern; callers relying on all-or-nothing (the shares coordinator) are unsafe on memory; type-name matching hides the carve-out from interface consumers. Fix: give memory a rollback buffer and remove the carve-out, or stop advertising `Transactor` atomicity for memory and replace the type-name match with a declared capability interface.
- **Safe-for-concurrent-use clause has no general concurrent-CRUD test** (`store.go:311-312`) — the only concurrent scenarios are domain-specific (ObjectID quiesce race `objectid_lookup.go:170`, INV02 refcount fuzz). No scenario runs concurrent `PutFile`/`GetFile`/`SetChild`/`DeleteChild`/`ListChildren` on overlapping handles under `-race`, which is what catches a map-write race or lost `SetLinkCount` update in the dir/parent/linkcount paths. Most-likely-to-differ clause across backends (memory mutex vs badger txn vs postgres isolation), essentially unverified for core CRUD. Fix: add a race-oriented scenario (N goroutines create/delete distinct children + readers) and rely on CI `-race`.

## 5. LOW findings (terse, grouped)

**postgres:**
- `Put` ON CONFLICT unconditionally sets `hash = EXCLUDED.hash` incl. NULL (`store/postgres/objects.go:105-112,463-470`) — a zero-hash Put would NULL a good hash (the #838 class, reversed). Safe today (both engine Put sites guard zero-hash) but defense-in-depth gap. Fix: `hash = COALESCE(EXCLUDED.hash, file_blocks.hash)`.
- `DeleteFile` swallows `link_counts`/`parent_child_map` delete errors (`store/postgres/transaction.go:357-358`) — `_, _ = tx.tx.Exec(...)` hides serialization/timeout failures and skips the retry path. Largely redundant (both cascade on `files` delete). Fix: propagate via `mapPgError`, or drop the redundant explicit deletes.
- Stale `file_type` comment in initial migration (`store/postgres/migrations/000001_initial_schema.up.sql:22,41`) — comment is 1-indexed + mis-ordered vs the 0-indexed `file_types.go:130` enum; the `BETWEEN 0 AND 6` constraint is actually correct. Doc-only. Fix: correct the comment.
- Hand-rolled `contains()`/`findSubstring()` mislabeled "case-insensitive" but case-sensitive (`store/postgres/errors.go:179-196`) — dead-weight reimplementation of `strings.Contains`, used only at `errors.go:67`. Fix: replace with `strings.Contains` and delete.

**memory + badger:**
- Memory `sortedDirCache` is dead code — read/invalidated but never written (`store/memory/store.go:561-563,565-646`; `transaction.go:231,258`) — every `ListChildren` re-sorts; the two helper funcs are now identical. ~80 LOC of phantom optimization. Fix: delete cache + collapse helpers (prefer per "less is more"), or actually populate it under the write lock.
- Badger `SetFilesystemCapabilities` swallows encode+Set errors and mutates `store.capabilities` unsynchronized (`store/badger/transaction.go:1085-1093`) — failed persist looks successful; the struct-field write races unguarded reads in `GetFilesystemCapabilities`/`Meta` fallbacks (`-race` finding; memory mutates under global `mu`). Fix: log the errors, guard the field with a mutex/atomic.
- Badger `DeleteFile` leaves `c:<uuid>:*` child entries; memory deletes the children map (`store/badger/transaction.go:254-265`) — bounded because callers guarantee emptiness (UUIDs never reused), but a contract-level divergence. Fix: prefix-scan delete over `keyChildPrefix`, or document the empty-handle precondition.

**store-contract conformance:**
- CreateShare-duplicate `ErrAlreadyExists` not asserted by the *shared* suite (downgraded from a refuted HIGH; see §3) — all three backends guard it and postgres has a dedicated test, but memory/badger get no shared-suite assertion. Fix: add a duplicate-CreateShare scenario to `storetest`.
- Deprecated-by-design `GetFileBlock`/`ListFileBlocks` remain on `MetadataStore` until #14 (`store.go:354-369`) — deprecation-bridge methods widen the contract every backend implements + the suite covers. Low (documented/tracked). Fix: complete #14, remove both; meanwhile add no new callers.
- `DeleteFile` no-check contract + nlink=0 orphan semantics not negatively tested (`interface.go:42-45,91-93`) — suite only deletes leaf files after manually removing the child entry. Fix: lock the no-check behavior + an nlink-to-zero round-trip via `GetLinkCount` across backends.

**metadata service / routing:**
- `MetadataServiceInterface` is dead code (`interface.go:20-183`) — 180 lines + a `var _` assertion are its only references; 1 impl, 0 type-callers, not a cycle-break. Pure duplicated documentation that drifts. Fix: delete (move docs onto concrete methods) or narrow to actually-mocked methods and use it.
- `LockFile` swallows `GetShareOptions` error and proceeds with nil options (`service.go:640-648`) — `if err != nil { shareOpts = nil }` then `calculatePermissions(nil...)` skips the ReadOnly-share branch and share-level squashing → fail-open. Low (WRITE re-checks read-only) but a swallowed error in a permission method. Fix: fail closed, or log at Debug and treat as read-only.

**file / dir / IO ops:**
- `ValidateName` does not reject `/` or NUL (`validation.go:129-145`) — slash rejection lives only in protocol handlers; per invariant 1 the central guard belongs here. A caller reaching `CreateFile`/`Move`/`CreateHardLink` without the per-handler filter concatenates `a/b` or `../x` into `File.Path` via `buildPath`, corrupting the Path-keyed postgres namespace (routing is UUID-based so unaffected; Path feeds snapshot/restore + descendant rewrite). Fix: reject `/` and `\x00` in `ValidateName`.
- `FlushPendingWriteForFile` re-caches a `CachedFile` with stale Size after flush (`io.go:399-401`) — re-caches the pre-write Size while the store holds `MaxSize`; only consumed by the soft-quota delta (`io.go:127-138`) → conservative quota rejection, no data impact. Fix: update `CachedFile.Size`/Mtime/Ctime/Mode before re-caching, or invalidate and reload.
- Cookie FNV64 hash collision can corrupt resume within an unchanged-dir window (`cookies.go:38-49`) — `dirHandle||name` keys the reverse map; a 64-bit collision between two live same-dir entries makes `GetToken` skip/repeat an entry. Probability negligible; mtime verifier mitigates most. Fix: detect collision on store, or derive the token deterministically from the entry (no reverse map).

## 6. Verified-correct (checked, no finding)

**postgres** — SQL fully parameterized ($N placeholders everywhere; the only `fmt.Sprintf`'d SQL is backup/restore table names validated against the hardcoded `backupTables` allowlist via `isKnownTable`, and ListPending placeholder *indices* with values still bound). `ensureStoreID` (#853) idempotent `INSERT...ON CONFLICT DO UPDATE COALESCE(NULLIF(...))` survives Reset/Restore truncation, preserves existing ULID, race-safe. `SetRollupOffset` (#853) monotone via `WHERE ... <= EXCLUDED` guard → `RowsAffected()==0` + `ErrRollupOffsetRegression` on regression; MaxInt64 overflow guard. `Put` hash persistence (#838) writes whenever `!Hash.IsZero()` regardless of state; restore backfills NULL hashes from `file_block_refs`. `ref_count` correctly OMITTED from ON CONFLICT (atomic +1/-1 not clobberable). Tx-bound FileBlock mutations run on `tx.tx` (rollback honored). Pool helpers wrap every op with a 10s acquire timeout + Release on Scan/Close. `WithTransaction` retries only 40P01/40001 ×3.
