## 4. MED findings (defer as issues unless trivially co-fixable)

**tx-atomicity (2):**

- **MED-1 — `updateDescendantPaths` swallows per-child `PutFile` error and skips unreadable children → directory rename partially commits stale `File.Path`** (`pkg/metadata/file_modify.go:722`, also `:712-717`). PR-B3 #888 ("Move rollback") propagated every inlined Move op (§3) but the recursion that actually rewrites descendant paths still does `if strings.HasPrefix(child.Path, oldPrefix) { child.Path = …; _ = tx.PutFile(ctx, child) }` (`:722`, **verified verbatim on current tree**), and a `GetFile` error on a child is logged-and-`continue`d (`:712-717`). The function returns nil unconditionally, so `WithTransaction` commits the whole rename even when descendant Path-rewrites failed or were unreadable. **Why:** a directory rename can commit with a subset of descendants retaining the OLD Path while their parent moved — the exact stale-`File.Path` blast radius (Path-keyed postgres snapshot manifest/restore + future descendant rewrite) that the outer fix targeted, on all three backends. **Fix:** return the error from `tx.PutFile` at `:722` so a failed descendant rewrite rolls the whole rename back; decide whether a `GetFile` failure at `:712` should abort rather than silently continue. (confidence 88)

- **MED-2 — Badger `DeleteShare` + `updateDescendantPaths` batch all per-file writes into one `badger.Txn`; un-retried `ErrTxnTooBig` fails large ops only on badger** (`store/badger/shares.go:248-305` `deleteShareFiles`; `file_modify.go:697-739` over a badger tx). `deleteShareFiles` issues 4-5+ `txn.Delete` per file and `updateDescendantPaths` one `tx.PutFile` per descendant, all into the single `badger.Txn` held by `WithTransaction`. Badger accumulates pending writes in memory and returns `ErrTxnTooBig` past `maxBatchCount`/`maxBatchSize` (~78k entries / ~9.6MB default); `WithTransaction` (`badger/transaction.go:71`) retries only `ErrConflict`, **not** `ErrTxnTooBig`, so the raw error propagates and the op fails outright. Postgres/memory have no per-tx size cap → silent cross-backend divergence: a very large subtree rename (reachable via every NFS/SMB rename) fails on badger while succeeding elsewhere. `backup.go:205/268` already uses a chunked `WriteBatch`, showing the pattern is known. **Why MED not HIGH:** requires a very large fan-out; metadata `DeleteShare` is currently only reachable via conformance/REST, not the production `RemoveShare` path. **Fix:** chunk `deleteShareFiles` via `NewWriteBatch` (mirror `backup.go`) or document+test the ceiling; for `updateDescendantPaths`, map `ErrTxnTooBig` to a split/retry path or document the per-rename descendant ceiling on badger; add a large-fanout conformance scenario. (confidence 62)

**cross-backend-divergence (3):**

- **MED (cross-1) — `GetFileByPayloadID`: memory store-level path leaves `File.ID` empty while tx-level path and badger/postgres set it → returned-then-`PutFile` writes to the `uuid.Nil` key (latent corruption)** (`store/memory/files.go:371-402`, esp. `:391-394`; diverges from `memory/transaction.go:539-563`, `badger/files.go:55-130`, `postgres/files.go:416-447`). The memory STORE-level method builds `&metadata.File{ShareName:…, FileAttr: attr}` with `File.ID`/`File.Path` left zero; the memory TX-level routes through `buildFileWithNlink` which DOES set ID/ShareName/Path (`store.go:508-513`); badger returns the full decoded file; postgres scans `f.id`. So 3 of 4 set ID, memory store-level does not. A `GetFileByPayloadID` → `PutFile(returned)` read-modify-write (the natural idiom `PersistFileBlocks` uses on the tx path) would, on the memory store path, encode a handle from a `uuid.Nil` ID and write an orphan row at `<share>:00000000-…`, silently losing the mutation. The only live consumer (`coordinator.go:216` `GetPersistedBlocks`) avoids the trap solely via `if file.ID == uuid.Nil { return nil }` (`:238`) — **verified on current tree, with the comment "Memory's GetFileByPayloadID returns blocks inline but leaves ID zero"** — a hand-coded workaround for this exact divergence. `testGetFileByPayloadID` only asserts `got.PayloadID==payloadID`, never that `got.ID` is non-nil, and never round-trips returned→PutFile→re-Get. **Fix:** route memory store-level through `buildFileWithNlink` (or set ID/Path) so all four agree; add a store_surface assertion that `GetFileByPayloadID` returns a File whose ID is non-zero and whose `EncodeFileHandle` round-trips to a `GetFile`-resolvable handle on all three backends. (confidence 90)

- **MED (cross-2) — `GetFileByPayloadID("")` has three observable behaviors across backends (and a fourth within memory)** (`postgres/files.go:417-422`; `memory/files.go:384-396`; `memory/transaction.go:546-549`; `badger/files.go:69-104`). Postgres returns `StoreError{ErrInvalidArgument}`; memory store-level has no empty guard and matches any file with `Attr.PayloadID == ""` (the default for every fresh file → returns an ARBITRARY file by map order); badger store-level likewise returns the first scanned empty-PayloadID file; memory tx-level explicitly skips empty rows → `ErrNotFound`. Four code paths, three outcomes. **Why:** an empty content id reaching this method on memory/badger silently resolves to an unrelated file; a follow-up `PutFile` corrupts a random file's metadata. `store.go:108` does not specify empty-input semantics and no scenario passes `""` — same conformance-blind class that hid #853/#838. **Fix:** specify empty-PayloadID semantics in `store.go` and enforce uniformly (reject `""` with `ErrInvalidArgument` everywhere, matching postgres, or skip empty rows everywhere matching memory-tx); add a store_surface assertion. (confidence 88)

- **MED (cross-3) — `DeleteChild` on a missing entry: memory/badger return `ErrNotFound`, postgres returns nil (idempotent) — contract docstring says `ErrNotFound`, untested** (`postgres/transaction.go:462-486`, no `RowsAffected` check; vs `memory/transaction.go:368-380`, `badger/transaction.go:345-371`). Every backend's docstring reads "Returns ErrNotFound if name doesn't exist" (e.g. `memory/files.go:201`, `postgres/files.go:126`); memory/badger honor it, postgres deliberately does not (comment cites the CASCADE-on-DeleteFile race). delete-then-delete-again yields `ErrNotFound` on memory/badger but nil on postgres; no scenario asserts `DeleteChild(missing)`. **Why:** a caller relying on the documented `ErrNotFound` to detect a stale/already-removed entry gets divergent control flow after a backend swap. Latent today (all callers DeleteChild a name they just confirmed exists). **Fix:** decide the contract — make it idempotent everywhere (drop the `ErrNotFound` clause + memory/badger) or have postgres check `RowsAffected`; add a `DeleteChild(missing)` store_surface assertion across all three. (confidence 84)

**handle-routing (1):**

- **MED (handle-1) — Share name >27 bytes overflows the 64-byte handle cap: memory backend PANICS, badger/postgres return a clean error; no upstream length validation** (`store/memory/store.go:541-553`, `panic` at `:552` — **verified on current tree**; reached from `CreateRootDirectory` `shares.go:234`, `GenerateHandle` `shares.go:22`, `CreateShare` `transaction.go:617`). A handle is `<shareName>:<uuid>` hard-capped at 64 bytes (`types.go:42-57`); a 36-byte UUID + colon leaves 27 bytes for the name, so e.g. `/projects/engineering/team-data` (31 bytes → 68-byte handle) over-caps. Memory's `generateFileHandle` treats the encode error as impossible and `panic`s (the code at `store.go:523` even documents the "share name length + 37 bytes" math) where badger/postgres surface a returned error. Reachable from REST `AddShare` → `CreateRootDirectory`; **no length check anywhere upstream** — `AddShare` (`runtime/shares/service.go:323-342`) validates only non-empty + `:` (added by #869, right where a length check belongs), and the REST `createShare` handler validates store IDs but not name length. **Why MED not HIGH:** the panic is caught by chi `middleware.Recoverer` (`api/router.go:61`) → request returns 500, process survives; the share simply fails to create on all backends — but identical operator input yields backend-dependent behavior (panic+spurious stack trace on memory vs graceful error on badger/postgres) and the suite has no long-name scenario, so it stays green. **Fix:** add an upstream length guard at `AddShare` next to the existing `:` check (reject `len(name) > 27` with `NewInvalidArgumentError`) and/or REST; independently, change memory `generateFileHandle` to return an error instead of panicking; add a long-name CreateShare/CreateRootDirectory conformance scenario asserting the same typed error across all three. (confidence 90)

**service-lifecycle (1):**

- **MED-4 — AddShare FAILURE / concurrent same-name loser leaks the metadata-service registration (lock manager + armed grace timer + dirChangeNotifier) — `cleanupShare` never calls `RemoveStoreForShare`** (`pkg/controlplane/runtime/shares/service.go:366-395`: `cleanupShare` 366-374, Phase-3 register 376-380, Phase-4 dup-reject 384-389). Phase 3 `RegisterStoreForShare` publishes the per-share lock manager + dirChangeNotifier and (for LockStore-backed stores with unclean state) ARMS a grace timer + fires `OnLockGraceStart`. If Phase 4 fails (duplicate at `:385`, or — more reachably — a concurrent second `AddShare` for the same name losing the registry race), `cleanupShare` closes ONLY the BlockStore and releases the remote store; it never calls `metadataSvc.RemoveStoreForShare(config.Name)`. The metadata service then permanently retains `lockManagers[name]`/`dirChangeNotifiers[name]`/`stores[name]`/`quotas[name]` for a share that failed and is NOT in the registry — the exact unbounded-map-growth + orphaned-lock-manager-with-live-grace-timer leak round-1 fixed for the RemoveShare path, reappearing at the symmetric AddShare-FAILURE boundary. The orphaned grace timer (`lock/grace.go:142` `AfterFunc`) survives to fire `onGraceEnd` and can call the NFSv4 `GraceCoordinator` `OnLockGraceEnd` out of band. `AddShare` is not serialized at runtime (`runtime.go:353`) or HTTP (`shares.go:401`), and `prepareShare`'s duplicate check is explicitly optimistic (`service.go:407-413`), so the concurrent-loser path is genuinely reachable. **Why MED not HIGH:** slow resource/timer leak + stale-routing hazard, not immediate data-loss/crash. **Fix:** in `cleanupShare` (or on every Phase-4 failure return) call `metadataSvc.RemoveStoreForShare(config.Name)`, mirroring `runtime.RemoveShare`; capture `metadataSvc` in the `cleanupShare` closure (already an `AddShare` parameter). (confidence 80)

## 5. LOW findings (terse, grouped)

**tx-atomicity:**
- **`CreateHardLink`/`RemoveFile` swallow `GetLinkCount` error inside tx and default to a count that drives nlink wrong** (`file_create.go:138`; `file_remove.go:104-108`). `CreateHardLink` does `linkCount, _ := tx.GetLinkCount(...)` then `SetLinkCount(linkCount+1)`; `RemoveFile` does `if lcErr != nil { linkCount = 1 }`. On memory/badger `GetLinkCount` returns `(0,nil)` for a missing key, so this fires only on a genuine backend error (e.g. postgres timeout) — but when it does, CreateHardLink sets a fresh `1` on a file that may already have links (undercount → premature content deletion) and RemoveFile assumes `1` → drops nlink to 0. Narrow window (real error required); a swallowed error inside the atomic section the round-1 TOCTOU fix otherwise hardened. **Fix:** propagate the `GetLinkCount` error from both tx closures so a backend failure rolls back. (confidence 55)

**cross-backend-divergence:**
- **`ListChildren` populates `DirEntry.Attr.Blocks` on memory/badger but omits it on postgres — documented "benign" but untested cross-backend** (`postgres/files.go:282-295` intentionally skips `file_block_refs`; `memory/files.go:326` and badger carry Blocks inline). The same asymmetry exists for `GetFileByPayloadID` (postgres omits per `coordinator.go:211`), where the coordinator already carries a per-backend fallback (loads via `GetFile`-by-handle for postgres) — the tell that the contract is unspecified. No scenario asserts whether `DirEntry.Attr.Blocks` is loaded; a new caller reading it silently degrades on postgres. Low because the current hot-path consumer never reads it. **Fix:** document in `store.go` that `DirEntry.Attr.Blocks` and `GetFileByPayloadID`'s `File.Blocks` are NOT guaranteed loaded (callers re-read via `GetFile`); add a conformance note pinning the agreed contract. (confidence 80)

**service-lifecycle:**
- **`UnifiedLockView` subsystem is dead code — no production caller; round-1 listed it as a live per-share resource** (`pkg/metadata/service.go:43` field, `:547-569` Get/Set, `:297` delete in RemoveStoreForShare; `pkg/metadata/unified_view.go` whole file). `SetUnifiedLockView` has zero production callers; `NewUnifiedLockView` is never invoked outside its own file; `unifiedViews` is always empty and `GetUnifiedLockView` always returns nil; `RegisterStoreForShare` never populates it. The docstring claims "one per share alongside the LockManager" but nothing creates it; `RemoveStoreForShare:297` deletes from an always-empty map. **Fix:** delete `unified_view.go` + the two accessors + the field + its `make()` + the `delete()`, or wire it (call `SetUnifiedLockView` in `RegisterStoreForShare` when the store implements `lock.LockStore`) if cross-protocol visibility is intended. (confidence 85)
- **`SwapMetadataStore`/`OpenMetadataStoreAtPath`/`openPostgresAtSchema` are unwired restore scaffolding** (`pkg/controlplane/runtime/stores/service.go:114-137`, `:161-204`, `:334-343`). The documented caller `CommitSwap` does not exist anywhere; the live restore path uses in-place `backupable.Restore` (`snapshot.go:1389`) + `bs.ResetLocalState`, not a store swap; `openPostgresAtSchema` always returns `errors.New("postgres schema-scoped open is not yet wired in Plan 04...")` so `OpenMetadataStoreAtPath` can never succeed for postgres. **Fix:** delete the three until swap-based restore is implemented, or finish `CommitSwap`. If kept, the metadata SERVICE side must also be handled: a swap re-points `stores.registry` but `metadataService.stores` keeps the old per-share reference, so a future swap caller must re-`RegisterStoreForShare` or routing resolves the stale store. (confidence 78)
- **`RegisterStoreForShare` overwrites `s.stores[name]` before the lock-manager early-return → same-name re-register with a different store instance leaves the lock manager bound to the OLD store's `LockStore` (split-brain) — latent footgun** (`pkg/metadata/service.go:138-145`; unconditional write at `:139`, early-return at `:143-145`). The docstring advertises "Calling this again for the same share will replace the previous store," but only the store map is replaced — the lock manager binding is not (`lm.SetLockStore` is never called on the new store on the early-return path). **Not reachable in production today:** `prepareShare` resolves via `storeProvider.GetMetadataStore(config.MetadataStore)` which returns the SHARED instance keyed by backend-config name, so a duplicate `AddShare` re-registers the identical pointer and `:139` is a no-op. Defense-in-depth: a future store-swap restore handing a new instance for an existing name would split lock state from metadata state. **Fix:** on the early-return path, if `lockManagers[name]` exists and the store implements `lock.LockStore`, call `lm.SetLockStore(newLS)`; or reject re-register with a different instance; move the `s.stores` write to after the existence decision. (confidence 62)

## 6. Verified-correct (round-2 checked, no finding)

**tx-atomicity:**
- Memory `WithTransaction` snapshot is rollback-safe despite shallow `maps.Clone` of `store.files`: every tx method REPLACES map entries (`PutFile` installs a new `&fileData`, `DeleteFile` deletes keys), none mutates an existing `*fileData.Attr` in place; shares (deep-copied struct-by-struct) and children (inner maps deep-cloned) handle the in-place-mutated cases.
- Cross-component tx-binding seam sound: `metadataCoordinator.resolveStore` (`coordinator.go:64`) returns `metadata.TxFromContext(ctx)` embedding `blockstore.FileBlockStore`, so `GetByHash`/`IncrementRefCount`/`DecrementRefCount` route through the caller's tx when WithTx-bound; `PersistFileBlocks` (`coordinator.go:188`) opens its own `WithTransaction` (not nested inside another metadata tx) → no postgres nested-transaction deadlock.
- Postgres `DeleteShare` FK ordering correct: deletes the `shares` row first (`shares.root_file_id` → `files(id)`, no cascade) then `DELETE FROM files`, with `parent_child_map`/`link_counts`/`file_block_refs` cascading from `files(id)`; `freedBytes` applied via `pendingDelta` post-commit (`postgres/transaction.go:870-921`).
- Postgres `WithTransaction` retries only 40P01/40001 (deadlock/serialization) ×3, applies acquire/rollback/commit timeouts, applies `pendingDelta` only after a successful commit (`postgres/transaction.go:61-128`).

**service-lifecycle:**
- Per-share `lock.Manager` spawns NO background goroutine/ticker/`AfterFunc` of its own (`manager.go` has no `go func`/`NewTimer`/`NewTicker`); only `GracePeriodManager` owns a timer, stopped by `AbortGracePeriod` → `RemoveStoreForShare` dropping the manager + aborting grace is leak-free for the manager itself.
- Healthcheck/stats/`GetUsedBytes` under concurrent register/remove is safe: `GetShareUsage` (`runtime.go:448-462`) and `HealthcheckShare` (`runtime.go:301-336`) resolve the store under RLock, release, then call on the captured ref; `RemoveStoreForShare` only deletes the metadata-service map entry and does NOT `Close` the metadata store (stores are provider-owned, closed only at shutdown via `storesSvc.CloseMetadataStores`) → no use-after-close.
- Lazy getters (`GetStoreForShare`/`GetLockManagerForShare`/`GetUnifiedLockView`/`GetQuotaForShare`/`lockManagerForHandle`) are pure reads under `s.mu.RLock` with no map mutation; `notifyDirChange` captures the notifier under RLock, releases before invoking with a panic-recover installed first (`service.go:987-1021`); `stores.Service` registry race-safe for the lifecycle ops it exposes (CloseMetadataStores snapshots under RLock; SwapMetadataStore commits under write lock so readers see old-or-new, never nil).

**handle-routing:**
- Handle-as-opaque invariant holds: every `DecodeFileHandle` call site (NFS `nlm.go`, SMB `adapter.go:646`, runtime `shares/service.go`, metadata `service.go`) decodes ONLY to extract the share-name prefix for routing; no handler parses the UUID body or interprets internal structure. Handle stability-across-restart sound (persistent backends store the minted UUID; routing decodes the same share component).
