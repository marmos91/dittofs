**Architecture invariants: clean.** All six sub-audits confirm the CLAUDE.md Runtime rules hold:
- **Single entrypoint** — Runtime is a ~777-line delegation facade; every data op routes Runtime → `sharesSvc`; no business logic inlined into the facade.
- **Opaque handles** — handlers never parse; only `metadata.DecodeFileHandle` inside the shares service extracts the share name for routing.

- **WRITE order** — `metadataStore.WriteFile` → `GetBlockStoreForHandle` → `blockStore.WriteAt` is honored.
- **Layering** — verified NO import cycles (no sub-package imports the runtime package; `store`/`models` have no upward deps; runtime does not import `controlplane/api`). `RuntimeSetter(rt any)` / `clientDisconnecter` are legitimate documented cycle-breaks. `go build ./pkg/controlplane/...` and `go vet ./pkg/controlplane/runtime/` exit 0.

**Verdict: NEEDS-FIX (narrow) — one HIGH, otherwise PATCH-grade.** The composition, refcounting, snapshot/restore/GC orchestration, and the REST auth/DTO/store boundary are in genuinely strong shape. The single HIGH is a **share-removal state leak**: `Runtime.RemoveShare` tears down the shares registry and block store but never deregisters the five per-share maps (`stores`, `lockManagers`, `unifiedViews`, `dirChangeNotifiers`, `quotas`) on `pkg/metadata.MetadataService`. This is **the exact class #897/#907 was supposed to fix, and the fix is absent/reverted in this tree** — `RemoveStoreForShare` exists nowhere and there are zero `delete(s.<map>...)` statements in `metadata/service.go`. It is both an unbounded-growth leak (incl. a leaked `*LockManager` with a possibly-armed grace timer) and a cross-generation lock/grace contamination bug on same-name re-create. (This is the same item flagged as MED in the area-6 review from the metadata side; verified here independently and raised to HIGH given the re-create lock-correctness harm.)

**Theme**: the correctness backbone is sound (snapshot fail-closed hold enumeration, restore commit-marker ordering, refcount unwinding, layering). The defects cluster in **teardown/lifecycle symmetry** — removal paths that don't mirror their add paths (H1 metadata-map leak, the mount-tracker UMNT-removes-all, the mount/cookie unbounded maps) — and **defense-in-depth** on the REST surface (uncapped bodies, raw-error leakage). Exactly what "tests pass" misses, since in-process add-then-access tests never exercise the removal/teardown path.

## 2. HIGH findings (triaged, ranked by blast radius)

### Share-lifecycle / teardown symmetry

- **H1 — `RemoveShare` leaks ALL five per-share `MetadataService` maps + lock-manager state (no `RemoveStoreForShare`; #897/#907 fix absent in this tree).** `pkg/controlplane/runtime/runtime.go:380-383` (`Runtime.RemoveShare`) + `pkg/controlplane/runtime/shares/service.go:776-814` (`shares.Service.RemoveShare`) + `pkg/metadata/service.go:41-48` (the five maps). **What:** `AddShare` populates per-share state across two layers — `RegisterStoreForShare` writes `s.stores[name]` (`service.go:139`), `s.lockManagers[name]` (`:197`), `s.dirChangeNotifiers[name]` (`:200`); `Runtime.AddShare` additionally calls `SetQuotaForShare` (`runtime.go:364` → `service.go:526-530`) writing `s.quotas[name]`; `s.unifiedViews[name]` is written via `SetUnifiedLockView`. On removal, `Runtime.RemoveShare` only drains snapshots then calls `sharesSvc.RemoveShare`, which deletes only from its **own** registry (`service.go:786`), wipes the snapshot dir, closes the BlockStore, and releases the remote ref. **Neither calls back into `MetadataService`.** Grep confirms `RemoveStoreForShare`/`DeregisterStore`/`UnregisterStore` exist nowhere in `pkg/metadata`, there are zero `delete(s.<map>...)` statements, and the only removal-style method (`RemoveFileLocks`) is per-file, not per-share, and is not invoked by `RemoveShare`. **Why:** two concrete harms. (1) **Leak** proportional to `AddShare`/`RemoveShare` churn — five maps grow without bound and the share's `*LockManager` (armed with a live grace-timer goroutine via `newGraceAwareLockManager`/`EnterGracePeriod`) is never torn down. (2) **Cross-generation contamination on re-create** — `RegisterStoreForShare` early-returns when `lockManagers[name]` already exists (`service.go:140-145`, explicit comment "Lock managers are ephemeral and not replaced when re-registering"), so re-creating a previously-removed share **replaces** `s.stores[name]` but **keeps** the prior generation's `*LockManager` (its byte-range locks, grace state, `DirChangeNotifier` wiring) and the prior `unifiedViews`/`dirChangeNotifiers` entries. This is exactly the cross-generation lock/grace contamination #897/#907 set out to eliminate. **Verifier rationale:** this finding *corrects* an earlier sub-audit output produced under a transient tool-output failure that hallucinated `RemoveStoreForShare` as existing-and-called; every cited location was then independently re-read and the bug confirmed exactly as described (the 5 maps exist and are populated by Add; the removal path touches none of them; no deregister API exists; the early-return keeps the stale lock manager on re-create). Severity HIGH is appropriate because it is **both** a resource leak **and** a lock-correctness regression on re-create — not a pure leak. **Fix:** add `MetadataService.RemoveStoreForShare(shareName)` that, under `s.mu`, deletes all five maps **and** tears down the lock manager (abort/stop any armed grace timer to avoid a goroutine leak and a stray `onGraceEnd` sweep), then call it from `Runtime.RemoveShare` symmetric with `AddShare`'s Register+SetQuota. Add a test asserting all five maps return empty/not-found after `RemoveShare`, plus a re-create test asserting a **fresh** lock manager (no inherited locks/grace). Sequence the deregister so a concurrent lazy getter cannot resurrect the entry — see M-share-2 in §4.

## 4. MED findings (defer as issues unless trivially co-fixable)

**runtime core / routing (2):**
- **`GetBlockStoreForHandle` returns the `*engine.Store` after releasing the RLock; `RemoveShare` can `Close` it during an in-flight op (use-after-Close TOCTOU)** (`shares/service.go:1121-1138` resolve vs `:776-814` RemoveShare) — the WRITE/READ data path operates on `bs` with no lock held while `RemoveShare` captures the same `bs` under the write lock, releases, then calls `bs.Close()` outside the lock. If `engine.Store.Close` is not safe vs concurrent `WriteAt`/`ReadAt`, this is a use-after-close (partial/lost writes, torn reads, or panic). Safety could not be verified (it depends on engine internals outside this area). Fix: pin the store with an in-flight refcount under RLock and have `RemoveShare` drain before `Close`, or make `Close` drain + return a clean `ErrClosed` mapped to `NFS3ERR_STALE`/SMB equivalent; add a concurrent-RemoveShare-during-active-WriteAt `-race` test. *(Overlaps M-share-3 below — same root, two sub-audits found it independently.)*
- **`CheckNetgroupAccess` returns `(false, nil)` on share-lookup failure** (`netgroups.go:120-124`) — an unknown/renamed/partially-loaded share is reported as an ordinary silent deny with no error and no log, indistinguishable from a legitimate netgroup deny (every other branch returns an error or logs Warn). Operators get silent access denials with zero diagnostic signal on real config drift. Fix: log at Debug on the share-miss branch and/or return a wrapped `shares.ErrShareNotFound`.

**share-lifecycle (3):**
- **(M-share-2) register/remove TOCTOU: lazy lock-manager creation can resurrect `lockManagers[share]` after deregistration** (`pkg/metadata/service.go:130-197` + `:346-401`) — even once H1's `RemoveStoreForShare` lands, if `Runtime.RemoveShare` deregisters metadata first while the share is still in the shares registry, an in-flight op resolving the share can call a lazy getter (`GetLockManagerForShare`/`ForHandle`) and re-insert `lockManagers[share]` *after* the delete, re-introducing the H1 leak/contamination under concurrent load. `RegisterStoreForShare` already documents the analogous registration-side race (`:185-196`); the removal path has no equivalent guard. Fix: remove from the shares registry FIRST so lazy getters fail to resolve, or add a per-share "removing" tombstone, or re-run the 5-map delete after `sharesSvc.RemoveShare` returns; add a concurrent RemoveShare-vs-GetLockManager test.
- **(M-share-1) concurrent `AddShare(sameName)` runs `RegisterStoreForShare` before the registry-uniqueness recheck** (`shares/service.go:368-383`) — Phase-3 `RegisterStoreForShare` (which **unconditionally replaces** `s.stores[name]`, `metadata/service.go:139`) runs before the Phase-4 write-locked duplicate recheck (`:377`). Two racing `AddShare(name)` both register (last writer wins on `s.stores[name]`); the loser fails the recheck and runs `cleanupShare`, which releases its remote ref but does **not** deregister its metadata store. Net: `MetadataService.s.stores[name]` may point at the loser's store while the shares registry exposes the winner — a registry/metadata mismatch, masked today only because `GetMetadataStore` returns a shared pointer. Fix: reserve the registry name under the write lock before the side-effecting register; have the loser's `cleanupShare` also deregister; add a concurrent `AddShare(sameName)` test.
- **(M-share-3) `RemoveShare` closes the per-share BlockStore with no in-flight-op quiesce barrier** (`shares/service.go:777-805`) — same root as the core-routing TOCTOU above; `DrainAllBlockStores` exists but is not invoked on this path, so safety rests entirely on `engine.Store.Close` being concurrency-safe and the contract is undocumented/brittle. Fix: document + test `Close` safety, or add a mark-removing → drain → `Close` quiesce step.

**sub-stores / adapters / mounts / identity (2):**
- **NFS UMNT removes ALL of a client's mount records, not just the unmounted export** (`runtime.go:~530-531` `RemoveMount`→`mountTracker.RemoveByClient`; caller `internal/adapter/nfs/mount/handlers/umount.go:68`) — mounts are keyed `(protocol, clientAddr, shareName)` but a single UMNT maps to `RemoveByClient(clientAddr)`, dropping the tracking rows for **all** the client's other still-mounted exports. `ListMounts`/`dfsctl list-mounts` then report the client as having unmounted everything after one export unmount. (NFSv3 UMNTALL is the operation that *should* map to `RemoveByClient`.) Fix: have UMNT call a per-`(client,share)` removal; reserve `RemoveByClient` for UMNTALL / connection drop — `Record`/`Remove` already support per-share keys; only the Runtime/handler wiring needs the share threaded.
- **Mount tracker has no staleness eviction — dead NFS clients leak entries forever** (`mounts/service.go:33-95`) — entries are removed only by explicit UMNT or admin `RemoveAll`; `MountInfo` records `MountedAt` but nothing sweeps. NFS MOUNT is stateless/connectionless, so a client that reboots/loses network never sends UMNT and its record persists indefinitely — the area-6 unbounded-map class made concrete. Fix: periodic stale sweep keyed on `MountedAt` (or NSM-style liveness), and/or wire NFS connection-close to `RemoveByClient` where a connection exists.

**snapshot / restore / GC (1):**
- **Hold-provider RLocks every share's delete-mutex for the full cross-share GC mark scan, serializing all snapshot deletes against any GC pass** (`snapshot_hold.go:64-71`) — `HeldHashes` RLocks every per-share RWMutex up front and holds them across the entire iteration (`ListSnapshots`/`os.Stat`/`streamManifest` for every snapshot of every share sharing the remote). `DeleteSnapshot` (`snapshot.go:1537-1539`) takes the WRITE side of the same per-share mutex for its whole sequence, so one GC mark phase blocks `snapshot delete` on **all** shares sharing that remote for the scan duration. Correctness is intact (it is the intended fail-safe so GC never reclaims a chunk whose manifest is mid-delete); the cost is latency, not deadlock/data-loss. Fix: acquire/release each share's RLock around only that share's manifest enumeration (move the lock inside the per-share loop body at `snapshot.go:73`) rather than locking the union up front — the delete path already takes one share lock at a time, so no ordering inversion is introduced.

**api / dto / models / store (2):**
- **Request bodies decoded with no size cap (memory-DoS) across all handlers** (`internal/controlplane/api/handlers/helpers.go:13-19` `decodeJSONBody`; also `snapshot.go:99`, `block_gc.go:90`, `blockstore.go:61`) — `json.NewDecoder(r.Body).Decode(v)` with no `http.MaxBytesReader`; `ReadTimeout=10s` (`config.go:70`) bounds slow-loris but not total body size, so a fast large body buffers into memory. MED not HIGH because every body-reading endpoint is behind `JWTAuth`+`RequireAdmin` (login/refresh read tiny uncapped structs). Fix: wrap `r.Body` with `http.MaxBytesReader(w, r.Body, ~1MB)` in `decodeJSONBody` and the three ad-hoc decoders; return 413 on overflow; optionally `DisallowUnknownFields()`.
- **Raw internal error strings leaked into HTTP response bodies (inconsistent sanitization)** (`internal/controlplane/api/handlers/system.go:48,51`; also `blockstore.go:43,62,75`; `snapshot.go:100`; `metadata_stores.go:90`; `migrate_status.go:141,147,180`) — several handlers interpolate raw `err.Error()` into the client-facing Problem detail, diverging from the package's own established convention (`block_gc.go:102-106`, `blockstore_audit.go:80-82`, `mapSnapshotError` `snapshot.go:276-329` all strip the underlying err and log at Debug). Internal error text can embed filesystem paths, SQL fragments, or store-config internals. Admin-gating limits but does not eliminate exposure (errors transit logs/proxies). Fix: return a fixed operator-friendly message and log the full err server-side, matching the existing convention.

## 5. LOW findings (terse, grouped)

**runtime core / routing:**
- Netgroup hostname matching uses a process-global DNS cache (`sync.Once` singleton) with TTL-only staleness and a latent nil-deref in `matchHostname` (`netgroups.go:21-26,156,188-196`) — live path is guarded by `ensureDNSCache()`, but `matchHostname` dereferences `pkgDNSCache` directly (would nil-panic if ever called without the guard); 5-min positive TTL means a revoked PTR keeps granting access for the window; global singleton couples independent Runtimes / complicates test isolation. Fix: have `matchHostname` call `ensureDNSCache()` itself (or take the cache as a param); document the TTL-bounded staleness.
- `LoadSharesFromStore` warn-and-skips per-share `AddShare` failures → silent partially-loaded server (`init.go:219-235`) — only `ErrLegacyLayoutDetected` aborts boot; any other failure (unresolved metadata store, invalid block-store config) is logged at Warn and the loop continues, so the process starts "successfully" while silently not serving a configured share. Fix: surface an aggregate "N shares failed to load" degraded-health signal (or non-zero exit); elevate the per-share log to Error.
- `InitializeFromStore` loads metadata stores but not shares; the required two-step boot ordering is implicit and unenforced (`init.go:20-27` vs `:149-238`) — `LoadSharesFromStore` is a separate exported function the caller must invoke afterward (shares depend on metadata stores being registered first); calling out of order or omitting it yields a runtime with zero shares and no error. Conflicts with the single-entrypoint invariant. Fix: fold share loading into `InitializeFromStore`, or document the order on both functions + add a boot-sequence ordering test.

**share-lifecycle:**
- `settings_watcher` invokes SMB-settings callbacks with no recover/timeout (`settings_watcher.go:252-258`) — a panicking callback kills the poll goroutine, a blocking one stalls it; either way all NFS+SMB settings hot-reloads silently stop server-wide. Fix: guard each callback with `recover()`; consider dispatch on a goroutine / with a timeout.
- `GetShare` returns the live `*Share` by pointer; `UpdateShare` mutates fields in place → torn multi-field reads during an update (`shares/service.go:816-853` UpdateShare + `:968-977` GetShare) — a handler caching the pointer can observe `ReadOnly`/`RetentionPolicy`/`RetentionTTL` half-applied (e.g. a write admitted because `ReadOnly` read false mid-update). The RWMutex guards the map, not the struct fields exposed by pointer. Low (rare admin action). Fix: return a value snapshot from `GetShare`, or expose lock-guarded accessors.

**sub-stores / adapters / mounts / identity:**
- SMB never records mounts — tracker / `ListMounts` is NFS-only despite the "unified across all protocol adapters" doc (`mounts/doc.go`; producers only at NFS `mount.go:160`/`umount.go:68`/`dump.go:71`) — `RemoveAllByProtocol("smb")`/`ListByProtocol("smb")` always return empty; observability/force-disconnect built on the tracker silently ignore all SMB clients. Fix: wire SMB session/tree-connect + logoff to `Record`/`Remove`, or scope the doc/types to NFS until SMB is wired.
- Two parallel squash implementations (`identity/service.go:29-64` `Service.ApplyIdentityMapping` vs `pkg/metadata/auth_identity.go:254` package-level `ApplyIdentityMapping`) — two code paths encode the same security-relevant squash/anon policy and can drift; a fix to one may miss the other. Fix: designate one canonical and have the other delegate. *(Semantics themselves verified correct — see §6.)*
- `stores.CloseMetadataStores` swallows close errors (log-only) and never clears the registry (`stores/service.go:81-95`) — `lifecycle.shutdown` cannot detect a failed backend flush/close, so shutdown appears clean even when a postgres-pool drain or badger close errored, masking durability problems. Fix: return `errors.Join` of close failures; optionally clear the registry.
- `adapters.EnableAdapter` returns a hard error when the adapter is already running (`adapters/service.go:133-149`, `startAdapter` "already running" `:171`) — an idempotent control-plane action errors even though the desired post-state (enabled + running) already holds, breaking safe retries. Fix: treat already-running as success after the config is persisted (check `IsAdapterRunning` first or swallow the already-running error).

**snapshot / restore / GC:**
- `DeleteSnapshot` deletes the DB row before wiping the on-disk dir; a dir-wipe failure strands manifest+dump on disk while the hold is already gone (`snapshot.go:1551-1580`) — `store.DeleteSnapshot` (row gone, `:1551`) precedes `os.RemoveAll(dir)` (`:1576`); on `RemoveAll` failure the row is deleted but `manifest.hashes`+`metadata.dump` persist, and since holds are keyed on `ListSnapshots` rows + on-disk manifest, a row-less stranded manifest is never enumerated (no retry handle). Not a data-loss path (post-delete the blocks are intentionally GC-eligible); it is on-disk file orphaning. Fix: `RemoveAll` BEFORE `store.DeleteSnapshot` so a wipe failure leaves the row+hold+retry handle intact (matches the dir-first create path), or log the orphaned path at WARN + add a startup reconcile.

**api / dto / models / store:**
- pprof endpoints unauthenticated when enabled; grace status unauthenticated (`pkg/controlplane/api/router.go:71-88` `/debug/pprof/*`, `:99-102` `/api/v1/grace`) — pprof (off by default) exposes heap/goroutine/CPU profiles (memory + DoS surface) with no auth when the `Pprof` knob is on. Fix: document that `Pprof` must bind loopback/admin-only, or gate it behind `JWTAuth`+`RequireAdmin` even when enabled.
- Admin password-reset leaves regular users' `MustChangePassword=false` (`internal/controlplane/api/handlers/users.go:305-309`) — documented deployment policy, not a defect; admin-chosen passwords for regular users are never force-rotated, so admins retain knowledge of standing user credentials. Flagged for security-reviewer awareness. Fix: set `MustChangePassword=true` for all roles on admin reset if org policy requires user-owned credentials.
- Snapshot show derives on-disk paths from URL share name; `normalizeShareName` does not reject path separators (`internal/controlplane/api/handlers/snapshot.go:70` + `:244-256`; `shares.go:27-39`) — name flows from a URL param into `os.Stat`/`os.Open` of `MetadataDumpPath`/`ManifestPath`; `normalizeShareName` only trims/re-prefixes a slash and does not strip interior `..`/separators (unlike `migrate_status.go`'s `isValidShareName`). Read-only, admin-gated, and gated by a prior share-existence lookup, so exploitability is low — but it lacks the defense-in-depth name validation `migrate_status.go` already applies. Fix: apply an `isValidShareName`-style guard (reject `..`/`/`/`\`/NUL) at the boundary; confine derived paths under the resolved `localStoreDir` via a `filepath.Rel` containment check.
- Netgroup in-use check uses a substring `LIKE` match on the config JSON blob (`pkg/controlplane/store/netgroups.go:42-50` DeleteNetgroup, `:119-125` GetSharesByNetgroup) — `config LIKE '%'+netgroup.ID+'%'` against `share_adapter_configs.config` is a raw substring match, not a structured JSON query → potential false positives (blocking a legitimate delete) or, less likely with full UUIDs, false negatives. Robustness nit; UUID-substring collisions improbable. Fix: parse the NFS adapter config and compare the `netgroup_id` field explicitly, or use a normalized join table.

**structural / design / resource:**
- Deprecated `NFSClientProvider`/`SetNFSClientProvider` shims retained against the pre-1.0 delete-eagerly policy (`runtime.go:724-728`) — 1-line passthroughs to the generic `adapterProviders` map; pure public-surface debt with no external consumers. Fix: migrate the few callers to `SetAdapterProvider("nfs", ...)`/`GetAdapterProvider("nfs")` and delete both methods.
- `openPostgresAtSchema` is a permanent-looking stub returning an error containing GSD plan IDs (`stores/service.go:334-343`) — unconditionally returns `errors.New("postgres schema-scoped open is not yet wired in Plan 04; Plan 06 ...")`; `OpenMetadataStoreAtPath` routes postgres restore through it (`:191-199`), so schema-scoped postgres restore-to-temp-schema is non-functional, and the message **leaks "Plan 04"/"Plan 06" GSD IDs into a runtime error** (forbidden by the no-phase/plan-IDs-in-source convention). Fix: wire the real `search_path`-scoped construction, or return a clean operator-facing error without plan IDs; confirm whether any live path reaches this before v1.0.
- Runtime facade is large (777 lines); verify no dead re-exports/passthroughs (`runtime.go:1-777`) — almost entirely thin delegators + type aliases (justified by the mandated single-entrypoint facade; **no split needed**); the only latent risk is accumulated unused exports. Fix: run `staticcheck U1000` + a per-export grep, drop zero-caller methods/aliases.

## 6. Verified-correct (checked, no finding)
