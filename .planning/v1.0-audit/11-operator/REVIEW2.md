## 2. HIGH findings (new this round, verified, ranked by blast radius)

### Deploy-path total liveness loss (operator never starts)

- **NEW-H2 — Webhooks default ON but no cert is wired in EITHER the Helm chart OR kustomize `config/default` → controller-runtime `certwatcher` read-error at startup → `os.Exit(1)` → operator CrashLoopBackOff on every documented install path.** `cmd/main.go:85` (`--enable-webhooks` defaults `true`), `:112-128` (webhook server built, no `GetCertificate` TLSOpt), `:198-203` (validating webhook registered, `failurePolicy=Fail`), `:216-218` (`mgr.Start` err → `os.Exit(1)`); `chart/values.yaml:7-9` + `chart/templates/deployment.yaml` (args = metrics/leader-elect/health-probe only, `readOnlyRootFilesystem:true`, no cert volume, no `--enable-webhooks=false`); `config/manager/manager.yaml:64-65,96-97` (same args, empty `volumes`/`volumeMounts`); `config/default/kustomization.yaml:19-21,49-53` (`[WEBHOOK]`/`[CERTMANAGER]` commented out). **What/why**: when webhooks are enabled and `--webhook-cert-path` is unset, the webhook server falls back to controller-runtime's default `CertDir` (`<tmp>/k8s-webhook-server/serving-certs`, `webhook/server.go:143-144`); it is registered as a non-leader-election runnable started synchronously at `mgr.Start` (`manager/internal.go:270-279,427`); `DefaultServer.Start` (`server.go:201-210`) calls `certwatcher.New` → `ReadCertificate` → `os.ReadFile` (`certwatcher.go:74,193`) on the **missing** cert and returns the error, which propagates to `mgr.Start` → `main.go:217 os.Exit(1)`. Neither the Helm chart (no cert-manager `Certificate`/`Issuer`, no webhook `Service`, no `ValidatingWebhookConfiguration`, no cert volume; `readOnlyRootFilesystem:true` also forecloses any `/tmp` fallback — and controller-runtime has none anyway) nor the kustomize default path provisions a cert. **Blast radius**: a clean `helm install ./chart` (`docs/INSTALL.md:73`) or `kubectl apply -k config/default` (`README.md:54`) puts the manager into CrashLoopBackOff **before any `DittoServer` can be reconciled** — total liveness loss on the documented happy path. `INSTALL.md:19` itself says cert-manager is "Required if using admission webhooks" yet neither path provisions it. *Verifier: confirmed end-to-end against the actual controller-runtime call chain in the module cache. Two immaterial doc nits in the source findings — pinned version is `v0.22.4` (not v0.22.5) and certwatcher lines are 63/193 — do not change the mechanism; HIGH retained. Reported independently and identically by two sub-audits (crd-webhook-config, helm-deploy-integration).* **Fix**: either (a) default `--enable-webhooks=false` in `chart/values.yaml` args and `config/manager/manager.yaml` args until cert wiring ships, OR (b) ship the full webhook stack — uncomment `[WEBHOOK]`+`[CERTMANAGER]` in `config/default/kustomization.yaml`, add a cert-manager `Certificate` + webhook `Service` + `ValidatingWebhookConfiguration` + cert `volumeMount` (`--webhook-cert-path`) to BOTH `chart/templates` and `config/`. Add a `helm template | kubeconform` render test and a `kind` smoke deploy that asserts the manager pod reaches Ready. **Note**: the documented escape (`ENABLE_WEBHOOKS` env var) does NOT work — see M-CFG-WEBHOOK below — so option (a) must use the flag, not the env var.

### Deletion/finalizer liveness block (CR hangs in Terminating forever)

- **NEW-H1 — Finalizer removal always 409-conflicts: the `Phase=Deleting` status write bumps `resourceVersion`, then the finalizer-removal `Update` runs on the now-stale in-memory copy; even the 60s force-removal escape hatch shares the stale object → `DittoServer` can hang in `Terminating` permanently, blocking owner-ref GC of all children.** `internal/controller/dittoserver_controller.go:468-470, 481-491, 503-504` (object fetched at `Reconcile:112`). **What**: `handleDeletion` does `dittoServerCopy := dittoServer.DeepCopy()` (468) then `r.Status().Update(ctx, dittoServerCopy)` (470) to set `Phase="Deleting"`. A status-subresource `Update` increments the **shared** `metadata.resourceVersion` (single etcd record), and controller-runtime decodes the new RV back **only into `dittoServerCopy`** (typed client `UpdateSubResource`) — so the original `dittoServer` keeps its stale RV from the line-112 Get. The subsequent `controllerutil.RemoveFinalizer(dittoServer,…)` + `r.Update(ctx, dittoServer)` (503-504) therefore sends a stale RV → apiserver returns 409 Conflict → `Reconcile` (118-121) returns the error → requeue. Crucially, line 470 runs on **every** deletion reconcile *before* the `elapsed>cleanupTimeout` check at 481, so the force-removal `Update` at 488 uses the same stale `dittoServer` and **also** conflicts — the 60s escape hatch does not escape. Each requeue re-Gets a fresh RV at 112 but 470 immediately re-invalidates it before 503/488 — self-perpetuating. (`performCleanup` at 495/514-579 never `Update`s `dittoServer` itself, only `pgCluster`, so the in-memory RV is never refreshed.) The only clearance is a manual `kubectl patch --type=merge -p '{"metadata":{"finalizers":[]}}'`. The intermittent-looking mask: if 470 *fails* it is swallowed at 472 and `dittoServer` stays fresh — which is exactly why it slipped past casual review. **Why**: liveness-blocking — owner-ref GC of the StatefulSet, Services, Secrets, NetworkPolicies, and (optionally) the `PerconaPGCluster` only runs after the finalizer is removed, so all children linger as long as the CR is stuck. Round-1 explicitly marked this path "verified-correct" ("force-removes the finalizer after 60s"), but the force path is broken by the same stale-RV. **No deletion test exists** (grep of `dittoserver_controller_test.go` finds only an add-finalizer reference at 271-274, no `DeletionTimestamp`/finalizer-removal test) despite the suite running envtest with real RV semantics that would have surfaced the 409. *Verifier: confirmed against the cited code and the resolved controller-runtime typed-client routing; bug exists exactly as described; HIGH retained.* **Fix**: do not reuse the line-112 object for the finalizer-removal `Update` after a status write — either (a) re-Get the `DittoServer` immediately before `RemoveFinalizer` at both 503 and 488, or (b) wrap the finalizer-removal in `retryOnConflict` (already defined at `controller.go:65`, which re-fetches/retries), or (c) `RemoveFinalizer` on `dittoServerCopy` (which now holds the fresh RV) / use `Patch` on a freshly-fetched object. Add an envtest deletion test (finalizer add → delete → assert finalizer removed and object gone) and a separate cleanup-timeout force-path test.

## 4. MED findings (terse, grouped by sub-area)

**reconcile-edges (deletion-path churn — amplifiers of NEW-H1)**
- **M-REC2-1 — `Phase=Deleting` status write re-issued on every deletion reconcile (incl. the cleanup-error 5s backoff loop), and that write is the mechanism that poisons the finalizer-removal RV.** `dittoserver_controller.go:468-473, 494-498`. The phase is already `"Deleting"` after the first write, so each subsequent write is a content-no-op that still bumps `resourceVersion` and invalidates the in-memory object used for finalizer removal in the same call. Fix: guard behind `if dittoServer.Status.Phase != "Deleting"` and/or use `r.Status().Patch`; best combined with the NEW-H1 fix (re-Get before finalizer removal).
- **M-REC2-2 — `performCleanup` leaves the in-server operator service-account undeleted when the admin Secret is already gone, and owner-ref children linger because GC only fires after finalizer removal (the broken step).** `dittoserver_controller.go:514-578`. `cleanupOperatorServiceAccount` returns nil on any failure (e.g. admin Secret GC-pending), so an `operator`-role user can be left provisioned on the `dfs` server after CR deletion (security residue); all owner-ref children linger while the CR is stuck (the dominant symptom is NEW-H1). Fix: fix NEW-H1 first so GC runs; separately, fall back to the auto-generated admin-credentials Secret name and emit a Warning event when the admin Secret is missing; add a test asserting the operator user is deleted (or an event emitted) on deletion.

**contract-drift-family**
- **M-CDF-1 — Operator-role token 403s on the adapters `settings` subtree; portmapper is never enabled on managed NFS servers.** `pkg/controlplane/api/router.go:286-316`. The operator's portmapper-enable GET and PATCH against `/adapters/nfs/settings` use the operator-role token, but the server gates that subtree behind `RequireAdmin`, so both 403. Cross-boundary authz drift round-1 missed. Fix: grant the operator role GET+PATCH on settings (or use the admin token); add a contract test.
- **M-CDF-2 — Operator emits `controlplane.jwt.issuer` but the server has no such field and hardcodes the issuer; viper silently discards the key.** `k8s/dittofs-operator/internal/controller/config/config.go:119` (see also M-CFG-ISSUER below — same root key, cross-reported). New config-schema drift surfaced as a CRD spec field with no round-trip test. Fix: drop `issuer` or wire it server-side; add a round-trip test.

**operator-security-deep**
- **M-SEC-2 — (downgraded from HIGH, §3) Managed `dfs` pod auto-mounts an unused namespace-`default` SA API token on the most network-exposed container.** `dittoserver_controller.go:943-1017`. Fix: set `AutomountServiceAccountToken: ptr.To(false)` on the managed PodSpec (the container needs no K8s API access); give a dedicated minimal-RBAC SA if a future feature ever needs it. Add a StatefulSet-builder test asserting automount is disabled.
- **M-SEC-3 — Managed `dfs` container has no secure-default `SecurityContext` and the webhook does not validate it (`allowPrivilegeEscalation` defaults true, no dropped capabilities); asymmetric with the hardened operator pod.** `dittoserver_controller.go:954` (`SecurityContext: dittoServer.Spec.SecurityContext`, no operator default; `getPodSecurityContext` 1036-1045 sets only `FSGroup`) + `api/v1alpha1/dittoserver_webhook.go:59-120` (no SecurityContext validation). MED not HIGH because the `dfs` image ships `USER 65532` (Dockerfile:59), so root is avoided by default — but a Pod-Security-Standards "restricted" namespace will **reject the operator-generated StatefulSet**, breaking the documented install on hardened clusters, and the operator's own pod is hardened (values.yaml drop ALL / `allowPrivilegeEscalation:false` / `readOnlyRootFilesystem` / `runAsNonRoot`). Fix: apply a secure default container SecurityContext when `Spec.SecurityContext` is nil (drop ALL caps, `allowPrivilegeEscalation:false`, `runAsNonRoot:true`, RO rootfs where feasible), merged with user overrides; optionally have the webhook warn on `privileged:true`.

**crd-webhook-config / helm-deploy-integration (deploy-contract drifts around NEW-H2)**
- **M-CFG-WEBHOOK — Documented `ENABLE_WEBHOOKS` (and `METRICS_BIND_ADDRESS`/`HEALTH_PROBE_BIND_ADDRESS`) env vars are bound to nothing; the only documented webhook-disable knob is a no-op.** `cmd/main.go:85` (and the 61-86 flag block — `flag.BoolVar`/`StringVar` only, no `os.Getenv`/`os.LookupEnv`/viper anywhere; grep confirms zero env reads) vs `docs/INSTALL.md:347-349`. Compounds NEW-H2: an operator following the docs to disable webhooks to escape the crash-loop sets an env var that is silently ignored, leaving the manager crash-looping with no obvious cause. Fix: bind the env vars in `main.go` (use `os.LookupEnv` as flag defaults) OR correct `INSTALL.md` to pass `--enable-webhooks=false` via Deployment args (and expose it as a chart value); add a knob-takes-effect test.
- **M-CFG-ISSUER — Operator renders `controlplane.jwt.issuer` into the ConfigMap but the server's `api.JWTConfig` has no `Issuer` field (it is hardcoded `"dittofs"` at `pkg/controlplane/api/server.go:69`); the key is silently discarded.** `k8s/dittofs-operator/internal/controller/config/config.go:119` + `config/types.go:52` vs `pkg/controlplane/api/config.go:48-62`. Same dead-config class as round-1 M-CFG-2 (`admin.username`) but a **distinct key round-1 did not enumerate**; `pkg/controlplane/api/README.md:157` also wrongly documents `issuer:` as working. (Cross-reported as M-CDF-2.) Fix: drop `Issuer` from the operator types/spec OR add it to `api.JWTConfig` and wire `server.go:69`; fold into the round-1 M-CFG-3 strict-decode round-trip test so all three dead keys (`cache`, `admin.username`, `jwt.issuer`) are caught at once.
