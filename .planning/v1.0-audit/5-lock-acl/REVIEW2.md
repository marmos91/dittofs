## 2. HIGH findings

One HIGH. It is a cross-protocol **liveness block** introduced as a side effect of the otherwise-correct #865 conflict fix.

### H-1. NLM blocking-lock waiter starves when the conflicting holder is SMB — SMB UNLOCK never drains the NLM blocked queue (liveness regression introduced by the #865 cross-scan)
**`pkg/adapter/nfs/nlm.go:183-184, 224-226, 336-403` (drain fires only on NLM unlock) — `internal/adapter/nfs/nlm/handlers/lock.go:159-208` (blocks on any conflict, incl. cross-protocol) — `pkg/metadata/lock/manager.go:810-840` (SMB `Unlock` has no NLM-waiter hook); SMB unlock entry `internal/adapter/smb/v2/handlers/lock.go:440` → `service.go:805` → `lm.Unlock`**

- **What:** After #865, `AddUnifiedLock` cross-scans `lm.locks` (`manager.go:1685-1695`, comment cites "area-5 H-3 / xproto H1"), so an NLM blocking LOCK (`req.Block`, i.e. Linux `F_SETLKW`) that overlaps an SMB-held byte-range lock now **correctly** returns `NLM4_BLOCKED` and the waiter is enqueued in `blockingQueue` (`nlm/handlers/lock.go:159-208`). That queue is drained **only** by `processNLMWaiters` (`nlm.go:336-403`), which is wired **only** to the NLM unlock callback — `onUnlock` fires at `nlm.go:183-184` inside `nlmService.UnlockFileNLM`, and `SetUnlockCallback` (`nlm.go:224-226`) launches `go s.processNLMWaiters(...)`. The SMB UNLOCK path (`lock.go:440` → `service.go:805` `UnlockFile` → `lm.Unlock` at `manager.go:810-840`) only mutates `lm.locks` and persists — it has **zero** hook into the drain. The Manager has break-callbacks for leases/oplocks but no byte-range-unlock notification. So when the SMB holder releases the blocking range, the NLM waiter is never re-attempted and the server never sends `NLM_GRANTED`. The NLM client waits for that server-driven callback (it does not poll), so it hangs until some unrelated NLM unlock on the same handle happens to fire the drain. `processNLMWaiters` re-attempts via `AddUnifiedLock` (`nlm.go:376`), which **does** cross-scan — so the grant would succeed if it were ever triggered. The fix is purely to also fire `processNLMWaiters` from the SMB UNLOCK path.
- **Why:** On a share exported over both SMB and NFS, an NFS client doing a blocking POSIX lock (`F_SETLKW`) over a range held by an SMB client blocks **indefinitely** after the SMB holder releases — a cross-protocol liveness block with no in-protocol workaround. This is strictly a side effect of the #865 conflict fix: before #865 the SMB lock was invisible to NLM so the NLM lock was granted immediately (the old data-corruption bug); now it correctly conflicts but the wakeup path was never extended cross-protocol, trading a correctness hole for a liveness hole. NFSv4 `LOCKW` is unaffected (it returns `NFS4ERR_DENIED` and relies on client retry — no server-side blocked queue); the gap is specific to NLM's server-driven GRANTED-callback model. `grep` confirms `processNLMWaiters`/`blockingQueue`/`GetWaiters` have **zero** references in `internal/adapter/smb/`, and no test covers NLM-blocked-by-SMB-then-SMB-unlock (the only `blockingQueue.Enqueue` test is the queue unit test in `internal/adapter/nfs/nlm/blocking/queue_test.go`).
- **Fix:** Fire `processNLMWaiters(handle)` from the SMB byte-range UNLOCK path so an SMB release re-drives blocked NLM waiters on the same handle. Cleanest seam: have `MetadataService.UnlockFile` (and `UnlockAllForOpen` / session teardown) invoke a registered post-unlock hook that the NFS adapter wires to `processNLMWaiters`, mirroring the existing NLM `onUnlock` callback; or have the lock `Manager` expose a protocol-agnostic "lock released on `handleKey`" notification that both the NLM drain and any future SMB blocked-lock waiter subscribe to. Add a `managertest`/integration case: an NLM blocking lock conflicts with an SMB lock, the SMB holder unlocks, and the NLM waiter is granted (GRANTED callback fired) without an intervening NLM unlock.
- **Verifier rationale (adversarial pass, confirmed REAL — HIGH retained):** Verified end-to-end at `/Users/marmos91/Projects/dittofs`. (1) NLM blocking LOCK on any conflict enqueues + returns `NLM4_BLOCKED` (`nlm/handlers/lock.go:159-208`). (2) The conflict can be cross-protocol because `AddUnifiedLock` cross-scans `lm.locks` (`manager.go:1685-1695`) — commit `82d62a3f` (#865), confirming the origin. (3) `processNLMWaiters` is the only drain and is wired only to the NLM unlock callback. (4) The SMB UNLOCK path has no hook into the drain (`lm.Unlock` only mutates `lm.locks` + persists). (5) `processNLMWaiters` re-attempts via the cross-scanning `AddUnifiedLock`, so a grant would succeed if triggered. (6) No test covers the scenario. HIGH is appropriate: indefinite client hang, no in-protocol workaround, introduced by the #865 correctness fix.

## 4. MED findings

Terse; grouped by sub-area. All are durability-asymmetry / recovery-edge defects the new persistence wiring introduced, or a latent cross-scan gap behind dead code.

**Lock persistence / recovery**

- **`SetLeaseEpoch` never persists the client-derived epoch — restored lease serves a stale (lower) epoch across restart, violating MS-SMB2 epoch monotonicity.** `manager.go:1324-1351` (`SetLeaseEpoch`, no `PutLock`); set from `internal/adapter/smb/v2/handlers/lease_context.go:496-507`; persist sites `store.go:359` / `leases.go:850`. `createAndGrantLease` persists `Epoch=1`; the handler then calls `SetLeaseEpoch(leaseKey, req.Epoch+1)` to align with the client's `RqLs`, but `SetLeaseEpoch` mutates `lock.Lease.Epoch` in memory for every matching record and issues **no** `PutLock` (unlike every other state-change path). If no break/upgrade re-persists before a restart on a persistent backend, `RestoreLocks` rebuilds the lease at the stale epoch and a later break dispatches a `LEASE_BREAK_NOTIFICATION` with a `NewEpoch` **below** the client's last-seen value (MS-SMB2 §2.2.14.2.11 / §3.3.4.7 require monotonic non-decreasing epoch). MED not HIGH: SMB clients normally re-establish leases via fresh `RqLs` on reconnect and `ReclaimLease` is unwired. Fix: have `SetLeaseEpoch` call `persistUnifiedLockLocked(record)` (it already holds `lm.mu`); add a grant→`SetLeaseEpoch`→`RestoreLocks` round-trip test asserting the high epoch survives.

- **NFSv4 delegations are granted and broken in-memory only — never persisted, so the cross-protocol delegation conflict surface is empty after restart.** `manager.go:2464-2504` (`GrantDelegation`, no persist) / `2642-2679` (`breakDelegations`, no persist). The persistence contract fully supports delegations (`PersistedLock` carries `DelegationID`/`DelegType`/`Deleg*`; `cloneLock` copies them; `FromPersistedLock` reconstructs the branch), yet `GrantDelegation` appends to `lm.unifiedLocks` with no `persistUnifiedLockLocked` and `breakDelegations` sets `Breaking=true` with no persist. Leases persist on grant; delegations never do — an asymmetry. After restart the manager has zero delegation records, so a delegation conflicting with a newly-granted SMB lease / NFS lock is undetected during grace. MED not HIGH: the NFSv4 `StateManager`'s own `DelegationState` is also lost on reboot and this impl does not reclaim delegations anyway, so clients re-OPEN and grace covers the window. Fix: either persist in `GrantDelegation`/`breakDelegations` (mirroring the lease paths), or explicitly document delegations as intentionally non-durable to stop the half-built machinery implying durability it does not deliver.

- **Lease persist paths use the cancellable request ctx (not `withPersistTimeout`) — a client disconnect mid-operation silently skips the durable write the code relies on.** `leases.go:454, 554, 586, 850, 940, 983, 1032` (`PutLock(ctx, …)`); contrast `manager.go:1166/1186` `withPersistTimeout`. The byte-range path derives a fresh bounded context via `withPersistTimeout()` precisely so request-ctx cancellation cannot skip a committed-in-memory write; the lease path passes the request ctx straight through, and every backend returns early on `ctx.Err()` (`memory/locks.go:291`, `badger/locks.go:57`, postgres). So a client disconnect during `RequestLease`/`AcknowledgeLeaseBreak` updates in-memory state but skips the durable write. Worse, the progressive-break next-stage persist at `leases.go:981-983` **swallows** the error (`_ = PutLock(ctx,…)`) while the comment at 976-980 explicitly states the persist must land or "a crash … would lose the second progressive stage on restart, leaving parked CREATEs to wait until the scanner timeout." Fix: route lease persist/delete through the same `withPersistTimeout()`-bounded context (or a shared `persistLeaseLocked` helper); at minimum log (not swallow) the error at `leases.go:940/983/1032`.

- **`RemoveFileLocks` / `RemoveFileUnifiedLocks` have no production callers and `RemoveFile` never reaps persisted locks — deleted-file lock records leak in the store and re-restore on every restart against a dead handle.** `manager.go:1001-1008` / `1824-1829` (in-memory delete only, no `DeleteLocksByFile`); `pkg/metadata/file_remove.go:27-163` (never invokes either). `RemoveFileUnifiedLocks` has zero callers; `RemoveFile` never calls `DeleteLocksByFile`. On file deletion, persisted byte-range/lease/delegation records for that handle are never reaped from badger/postgres, accumulate unbounded across delete churn, and are re-loaded by `RestoreLocks` on every restart against a handle that no longer exists (no handle-existence guard except dir-lease reclaim at `reclaim.go:65-68`). MED: resource leak + stale-restore, reaped only opportunistically on client disconnect / grace sweep. Fix: call `DeleteLocksByFile(ctx, handleKey)` (implemented on all three backends) from `MetadataService.RemoveFile` after the metadata tx commits — or wire the two methods to issue the persisted delete and invoke them from the delete path; otherwise delete the unused methods and document that reaping is disconnect/grace-driven only.

**Byte-range cross-protocol**

- **`UpgradeLock` (NLM/NFSv4 shared→exclusive) does not cross-scan `lm.locks` — bypasses the #865 cross-protocol exclusion (latent: dead production code).** `manager.go:1601-1658` (Step-2 conflict scan at `:1636` iterates only `lm.unifiedLocks`). Unlike `Lock` and `AddUnifiedLock` (both fixed by #865 to cross-scan the other map), `UpgradeLock` would grant a shared→exclusive upgrade even with an overlapping SMB byte-range lock in `lm.locks`, re-opening the exact hole #865 closed (MS-FSA §2.1.5). `grep` confirms `UpgradeLock` has **no** production callers — NLM/NFSv4 acquisition routes through `AddUnifiedLock` — so this is currently dead code and the gap is latent. Fix: add a cross-scan of `lm.locks` (via `fileLockConflictsWithUnified`) into Step 2 to match `Lock`/`AddUnifiedLock`, or delete `UpgradeLock` and its interface method as dead code (also reduces the round-1-flagged over-broad 48-method `LockManager` interface).

## 5. LOW findings

**Lock persistence / recovery**

- **Grace-end timer callback can run `RemoveClientLocks` against a store being closed by a concurrent `RemoveShare` (`AbortGracePeriod` does not cancel an in-flight `AfterFunc`).** `grace.go:142-144` (`time.AfterFunc`) / `184-207` (`exitGracePeriodInternal`); abort at `service.go:292`. `EnterGracePeriod` arms `gpm.timer = time.AfterFunc(duration, exitGracePeriodInternal)`, whose `onGraceEnd` closure (`service.go:417-429`) calls `lm.RemoveClientLocks(c)` → `lockStore.DeleteLocksByClient(ctx)` (`manager.go:2856-2862`). `AbortGracePeriod`→`timer.Stop()` (`service.go:292`) does **not** wait for or cancel an already-fired callback, so a concurrent `RemoveShare` can run `DeleteLocksByClient` against a badger DB being closed. Benign today: the error is logged-and-swallowed (`manager.go:2860`) and badger `Close` is idempotent (`09864d93`) — no crash, no data loss (the share is being removed). LOW: a genuine use-after-close window on teardown that could surface as confusing ERROR-log noise or a panic if a future backend `Close` is not panic-safe. Fix: have `AbortGracePeriod`/`Close` join any in-flight callback (running-flag + sync, or a done-channel select), or document that grace-sweep store errors during teardown are expected and keep every backend `Close` panic-safe.

**Grace-coordinator edges**

- **Refcount leak + stale first-in-wins comment in `RegisterStoreForShare` `lmExists` abort path (introduced by #919 refcount; latent because M2 serializes the only caller).** `service.go:211-238` (abort branch + comment `220-226`); coupled to `pkg/adapter/nfs/grace_coordinator.go:79`. The abort branch fires `graceCoordinator.OnLockGraceEnd()` only for `removedMidFlight`, never for pure `lmExists` (`service.go:235-236`); its comment asserts the loser's redundant `OnLockGraceStart` was a no-op under the OLD first-in-wins policy. PR #919 replaced first-in-wins with a refcount: `OnLockGraceStart` now does `c.active++` **unconditionally** (`grace_coordinator.go:79`). If two callers race `RegisterStoreForShare` for the same share and both entered grace, both increment (to 2); the `lmExists` loser skips `OnLockGraceEnd`, so the coordinator never reaches refcount-zero and never performs its lockstep `ForceEndGrace` — v4 grace is held until the hard-timer backstop instead of lifting in lockstep. **Unreachable today**: #919's own M2 share-name reservation (`shares/service.go` `reserveShareName` under `s.mu`) serializes `AddShare(sameName)`, the sole production caller. LOW: latent + stale comment that misleads the next editor; v4 hard timer bounds the blast radius (no indefinite wedge). Fix: either make `OnLockGraceStart` increment only on coupling paths so a redundant start is again a true no-op, or fire exactly one `OnLockGraceEnd` in the `lmExists` branch when `lm.IsInGracePeriod()`; update the `service.go:220-226` comment to refcount semantics; add a regression test simulating two `RegisterStoreForShare` calls for one in-grace share and asserting `active` returns to zero.

**ACL enforcement**

- **Share-write bypass grants write to any share-writable user on an allow-only DACL that does not grant that user (additive-semantics over-grant).** `pkg/metadata/auth_permissions.go:265-274`. `checkFilePermissions` bypasses the file ACL and grants `PermissionWrite`/`PermissionDelete` whenever `ctx.ShareWritable && !ctx.ShareReadOnly && !acl.HasExplicitDeny(file.ACL)`. `HasExplicitDeny` only checks for a DENY ACE *type* anywhere in the ACL — it does **not** check whether the allow-only ACL actually grants write to *this* requester. So `[ALLOW alice:WRITE_DATA]` (allow-only, no DENY) lets bob — a different share-writable user — write, because the bypass fires before `evaluateACLPermissions`. **Not an enforcement bypass in the malicious sense**: any DENY ACE disables the bypass entirely (conservative whole-ACL gate), SMB clients already passed a share-level + DACL CREATE gate, and this is documented-intentional additive semantics, load-bearing for smbtorture `stream-inherit-perms` / `create.multi`. The implication: on a writable share, file-level write restriction requires an explicit DENY ACE — allow-only DACLs do not restrict share-writable users at the per-op gate. Fix (lower-risk v1.0 choice): document this as a known limitation; or scope the bypass to fire only when `file.ACL == nil`, routing allow-only ACLs through `evaluateACLPermissions`.

## 6. Verified-correct

Checked this round and found OK (beyond the round-1/xproto HIGH re-verifications in §3).

**Byte-range cross-protocol unification (#865)**
- Acquisition cross-scans both ways and previews agree with acquire: `Manager.Lock` scans `lm.unifiedLocks` (`manager.go:761-767`), `AddUnifiedLock` scans `lm.locks` (`manager.go:1685-1695`); `TestLock` (929-933) and `TestUnifiedLock` (1739-1746) mirror the cross-scan.
- `Length==0` protocol clash handled correctly in the cross-scan helpers: SMB zero-byte locks always carry `IsZeroByte=true` (set at `smb/v2/handlers/lock.go:474`) and short-circuit in `fileLockConflictsWithUnified` (`manager.go:612`) and `CheckIOConflict` (`manager.go:549`); NLM/NFSv4 `Length==0` means to-EOF, handled via `RangesOverlap`/`rangeLast=maxUint64`.
- Single `sync.RWMutex` (`manager.go:680`) guards both maps; cross-scans run under one acquisition — no lock-ordering / deadlock hazard.
- SMB-blocked-by-NFS direction has **no** missed-wakeup: cross-protocol blocking SMB locks use poll-retry (`time.Ticker` `BlockingLockRetryInterval`) in both inline `acquireLockWithRetry` (`lock.go:644`) and async park (`lock_async.go:201`), re-attempting against the cross-scan surface — so an SMB lock blocked by an NFS lock is woken when the NFS lock releases. (The NLM-blocked-by-SMB direction is the H-1 gap — NLM uses a server-driven callback, not poll.)
- SMB lock-denial mapping is safe for cross-protocol conflicts: a denial caused by an NLM lock surfaces `StatusLockNotGranted` (`lock.go:497-504`); the conflicting lock's `OwnerID` (`nlm:…`) and `SessionID=0` are used only for logging — no parse, no panic.
- `ToPersistedLock` not setting `IsZeroByte` (`store.go:331-353`) is structurally harmless: `UnifiedLock` has no `IsZeroByte` field; SMB zero-byte locks travel the `FileLock` path (which round-trips it at `manager.go:1095, 1230`); NLM/NFSv4 never produce zero-byte locks. The xproto MED "unified persist path downgrades zero-byte" cannot fire.
- `fileLockConflictsWithUnified`'s "cross-protocol locks are always different owners" assumption is sound: SMB owner IDs are hex/`smb`-prefixed, NLM/NFSv4 are `nlm:`/`session:`/`nfs4:` prefixed, so `==` is never true across protocols; the helper correctly reduces to range-overlap + exclusivity and excludes leases/delegations.

**Grace coordinator (#919 + area-4 reboot-grace #904/#907/#909/#911)**
- v4 `GracePeriodState.startGrace` (`internal/adapter/nfs/v4/state/grace.go`) always arms a hard timer (`AfterFunc(duration, endGraceWithReason)`) on a non-empty roster; `endGraceWithReason` is idempotent via a state re-check — a genuine always-on backstop, so even a coordinator refcount leak (the §5 LOW) cannot wedge new-state creation indefinitely.
- `RemoveStoreForShare` double-decrement prevented: it captures `lm.IsInGracePeriod()` under `s.mu` **before** `AbortGracePeriod`; `AbortGracePeriod`→`Close` stops the timer and sets `state=Normal` **without** firing `onGraceEnd`; `exitGracePeriodInternal` re-checks `state==Normal` after acquiring `gpm.mu`. Exactly one `OnLockGraceEnd` per share window.
- `adapter.go` boot catch-up (`SetGraceCoordinator` then `OnLockGraceStart` for shares still in grace) is race-safe vs a share grace timer firing concurrently — either catch-up sees Active (start sequenced before timer end → balanced) or Normal (does not start; timer end hits `active==0` and is ignored by the underflow guard → balanced).
- `OnLockGraceEnd` underflow guard (`if c.active==0 return`) absorbs early/spurious ends (covered by `TestGraceCoordinator_EndUnderflowIgnored`); boot-seeded-roster path defers to the v4 machine's reclaim early-exit + hard timer rather than force-ending (covered by `TestGraceCoordinator_BootSeededRosterNotForceEnded`); `removedMidFlight` abort balance is net-zero under the refcount.

**ACL enforcement (round-1 0-HIGH verdict re-confirmed on current `develop`)**
- `acl.Evaluate` (`evaluate.go:170-272`) is fail-safe under multi-bit masks: returns true only when `(allowedBits & requestedMask) == requestedMask`, so a partial allow can never over-grant a multi-bit request (verified against `PermissionWrite`→`WRITE_DATA|APPEND_DATA`, `auth_permissions.go:426`).
- DENY-before-ALLOW / non-canonical DACL: "newly-decided-bits-only" accumulation (`evaluate.go:233/238`) + per-bit probe loops (`auth_permissions.go:753-761`, `ProbeBitsAll` `799-803`) make first-match-wins exactly correct; a trailing DENY cannot over-grant; an explicit DENY ahead of an ALLOW wins per bit. Matches Windows/Samba.
- OWNER_RIGHTS (S-1-3-4): pre-scan (`evaluate.go:196-212`) counts only ACCESS_ALLOWED/DENIED non-inherit-only OWNER_RIGHTS ACEs (excludes AUDIT/ALARM); when present and requester is owner, OWNER@ ACEs are made not-to-match (`306-308`), OWNER_RIGHTS ACEs match (`328`), and the §2.5.3.2 implicit owner grant is suppressed (`263`). A DENY OWNER_RIGHTS:WRITE_DAC correctly denies the owner. Spec-faithful per MS-DTYP §2.5.3 / Samba `se_access_check`.
- Owner-implicit grant (`evaluate.go:263-268`) is masked by `deniedBits` (`&^`) so an explicit DENY for READ_CONTROL/WRITE_DAC/WRITE_OWNER still wins; WRITE_OWNER gated on `RequesterHasTakeOwnership` (restricted to BUILTIN\Administrators S-1-5-32-544 and RID-500). `AnonymousFileOwnerUID` sentinel prevents anon UID==0 collapsing onto root-owned files' OWNER@.
- GENERIC_* expansion (`generic.go ExpandGenericMask`) applied at every documented boundary — SMB CREATE `DesiredAccess` (`create.go:206`), SET_INFO Security per-ACE (`security.go:566`), inherited non-INHERIT_ONLY ACEs (`inherit.go:283`), defensively in `computeMaximalAccess` (`create.go:1486/1493`); generic bits stripped post-expansion; GENERIC_ALL → `0x001F01FF`. NFSv4 has no GENERIC concept so `DecodeACLAttr` correctly does not expand.
- ACE order preserved verbatim through SET: `parseDACL` (`security.go:563-570`) appends in wire order; SD canonicalization (`security.go:483-499`) touches only AutoInherited/Protected control flags, never ACE order. DENY-before-ALLOW intent survives SET_INFO Security round-trips. `ValidateACL` (`validate.go:38-54`) intentionally does not enforce canonical ordering (correct per MS-DTYP §2.4.5).
- `HasExplicitDeny` (`evaluate.go:104-114`) counting INHERIT_ONLY DENY ACEs is fail-safe (disables the share-write bypass even for a non-applicable DENY — errs stricter).
- `ComputeInheritedACL` (`inherit.go:114-296`) preserves parent ACE order, applies INHERITED_ACE only when `parent.AutoInherited`, expands GENERIC_* on emitted non-INHERIT_ONLY ACEs, handles CREATOR dual-emission + NO_PROPAGATE, and enforces `MaxACECount` FIFO truncation before append (earlier/precedence parent ACEs win on overflow). Mirrors Samba `calculate_inherited_from_parent` + `desc_expand_generic`.
- AUDIT/ALARM ACEs are evaluation no-ops in `Evaluate` (`evaluate.go:241-243`) — even though `buildDACL` emits AUDIT ACEs into the SMB DACL body (the known round-1/xproto display-correctness MED), they are never subjected to internal access evaluation, so there is no enforcement bypass from the misplacement.
- `SynthesizeFromMode` is dead production code (zero non-test callers); `SynthesizeWindowsDefault` is display-only for SMB DACL when `file.ACL==nil` (`security.go:327`), never stored/enforced — the round-1 Administrators-FullControl flag is display-only; enforcement falls to POSIX mode via `CheckFileAccessWithParent`'s `file.ACL==nil` branch.
- `CheckFileAccessWithParent` (`auth_permissions.go:702-843`) per-bit probe + MS-FSA §2.1.4.13 semantics (FILE_READ_ATTRIBUTES always granted, ACCESS_SYSTEM_SECURITY→ErrPrivilegeRequired, parent FILE_DELETE_CHILD override, MAXIMUM_ALLOWED still requires explicit named bits) match Samba and the NFS-side root/anon handling.
- No new ACL enforcement fixes have shipped since round-1 — `git log` on `pkg/metadata/acl/` + `security.go` shows the last ACL commit is wave-0 #666; the 0-HIGH ACL verdict reflects current `develop`.
