<objective>
ADAPT-02 — Replace SMB regular-file READ's inline `make([]byte, actualLength)` with the pooled `common.ReadFromBlockStore` path; wire a nil-safe `ReleaseData func()` field through `SMBResponseBase` / `HandlerResult` so the encoder returns the pooled buffer to `pool.Put` AFTER the wire write completes (plain + encrypted + compound paths). Pipe and symlink READ variants are explicitly NOT routed through the pool — their buffer sources (`mfsymlink.Encode` output / `pipe.ProcessRead` return value) are already heap-allocated, and copying them into a pool-backed slice would be wasteful memcpy without benefit. Document this in-handler so future contributors know it's deliberate.

Purpose: Brings SMB regular-file READ to pool parity with NFS — eliminates per-request `make([]byte, n)` on the hot path and gives SMB the same proven Release-after-encode lifetime guarantee NFSv3 already has. D-09's "release after response encoding, not defer in handler" pattern ensures the pooled buffer stays valid through compound response assembly and encryption without any handler needing to know about framing timing.

Revised scope note (per checker): D-10's original wording "one release mechanism for all READ variants" is narrowed to "one release mechanism for the regular-file READ path". Pipe/symlink are structurally non-candidates for pool integration because:
- `handleSymlinkRead` sources its buffer from `mfsymlink.Encode(file.LinkTarget)` — the returned slice is already a heap allocation produced by the encoder. Wrapping it in a pool copy would allocate + copy + schedule a release for no observable benefit (symlink reads are uncommon and always ≤1067 bytes per MFsymlink spec).
- `handlePipeRead` sources its buffer from `pipe.ProcessRead(int(req.Length))`, which returns an internal PipeManager slice whose lifetime is owned by the pipe state machine (not the handler). Coercing it through the pool would either require a copy (memcpy overhead) or double-free risk (two owners: PipeManager and pool).

Both cases stay on the non-pooled path with `ReleaseData: nil` and a short in-handler comment explaining the decision.

Output: `SMBResponseBase` + `HandlerResult` grow a `ReleaseData func()` field; response encoder invokes it in one canonical place (after `WriteNetBIOSFrame`); compound handler invokes all collected `ReleaseData` closures once after the single composed write; SMB regular-file READ handler uses `common.ReadFromBlockStore` and hands `readResult.Release` to the encoder via `ReleaseData`; pipe/symlink READ variants remain on direct heap-allocated buffers with inline documentation; non-pooled commands continue to work with `ReleaseData == nil`. One atomic signed commit labeled `ADAPT-02`.
</objective>
