      func (bs *BlockStore) Flush(ctx context.Context, payloadID string) (*blockstore.FlushResult, error) {
          // File-level dedup pre-hook: if a fully-quiesced manifest matches
          // an already-stored ObjectID, we skip the upload pump entirely.
          if bs.coordinator != nil {
              specBlocks, blockStates, err := bs.syncer.snapshotPendingBlockRefs(ctx, payloadID)
              if err != nil { return nil, fmt.Errorf("snapshot pending blockrefs: %w", err) }
              if len(specBlocks) > 0 {
                  fileObjectID, err := bs.coordinator.GetFileObjectID(ctx, payloadID)
                  if err != nil { return nil, fmt.Errorf("get file objectID: %w", err) }
                  hit, err := bs.syncer.trySpeculativeFileLevelDedup(ctx, payloadID, specBlocks, fileObjectID, blockStates)
                  if err != nil { return nil, fmt.Errorf("file-level dedup: %w", err) }
                  if hit {
                      return &blockstore.FlushResult{Finalized: true}, nil
                  }
              }
          }
          // Delegate to syncer's mirror loop.
          return bs.syncer.Flush(ctx, payloadID)
      }

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| engine.Delete critical section | DecrementRefCount + DeleteSynced fire inside the same loop iteration — no other goroutine can re-Mark this hash between the two calls because the coordinator's refcount-zero state precedes our DeleteSynced. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-07-01 | Tampering | Parallel Syncer pass re-Marks a hash that engine.Delete just deleted | mitigate | Inside the same loop iteration: DecrementRefCount makes newCount==0 (coordinator-visible) BEFORE DeleteSynced fires. A parallel ListUnsynced pass that snapshotted before our Delete already captured the hash but reads from local.Get — if the local CAS chunk is already gone (DeleteChunk fired earlier in engine.Delete; verify ordering by reading the file), local.Get errors and the mirror loop surfaces the error rather than re-Marking. If the chunk survives momentarily, the Mark race is benign because our DeleteSynced cleans it up post-race. Document this in the engine_delete_test.go test 1 godoc. |
| T-18-07-02 | Denial of Service | Many concurrent Deletes thrash DeleteSynced backend | accept | DeleteSynced cost is O(1) per backend; Delete is already a low-throughput op (file delete). No new amplification. |
| T-18-07-03 | Repudiation | DeleteSynced error swallowed | mitigate | Logged at Warn with hash + error; never blocks Delete (orphan marker is benign). Operators can grep logs. |
| T-18-07-04 | Tampering | Coordinator returns wrong newCount (e.g., 0 when refcount actually > 0) | mitigate | Coordinator contract per coordinator.go godoc: "DecrementRefCount atomically decrements; returns the new count". Existing coordinator tests cover this. If newCount lies, DeleteSynced fires while other files still reference the hash → orphan synced marker → benign (next Put for that hash skips upload, but the bytes ARE on remote). Acceptable: synced marker drift is a hint, not authoritative. |
</threat_model>
