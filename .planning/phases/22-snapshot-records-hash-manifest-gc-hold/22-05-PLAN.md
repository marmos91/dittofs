<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Disk manifest files | Provider reads attacker-controlled bytes (in principle — a compromised filesystem could feed garbage). ReadManifest's parser is the mitigation. |
| GC sweep on the live remote | Per CONTEXT D-04 the on-disk manifest is the ground truth; a missing manifest for a ready row aborts the run (fail-closed) so the worst case is "GC does not run", never "live data deleted". |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-05-01 | Tampering | manifest.hashes file | mitigate | snapshot.ReadManifest enforces hex length + 32-byte parse via blockstore.ParseContentHash; malformed lines return ErrInvalidManifestLine which propagates up and aborts the GC run |
| T-22-05-02 | Denial of Service | manifest file read errors | mitigate | Errors surface as wrapped error with share + snapshot ID + path; GC abort preserves data (orphan-not-deleted is preferred) |
| T-22-05-03 | Information Disclosure | logger output | accept | Debug-level log lines include share, snapshot ID, count, remote endpoint ID — same data already logged by blockgc.go for the engine's own correlation keys |
| T-22-05-04 | Tampering | RemoveShare cleanup hook | mitigate | os.RemoveAll runs on a known per-share directory under localStoreDir; not user-controlled. Failed removal logs warn but does not interrupt the registry removal — orphaned files are harmless |
| T-22-05-SC | Tampering | go.mod additions | accept | No new third-party dependencies; all imports already in go.mod |
