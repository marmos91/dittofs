  ### Required when upgrading from v0.15.x or earlier
  
  v0.16.0 replaces the legacy `<share>/<file>/<idx>.blk` block layout with a content-addressed store (CAS). Pre-v0.16 storage directories must be migrated before `dfs start` will succeed.
  
  ### Boot-guard behavior
  
  On startup, `dfs start` checks each share's directory for a `.cas-migrated-v1` sentinel file at the share root. If the sentinel is missing AND legacy `.blk` files are present, the server refuses to start (per-share fail-fast):
  - Exits with code **78** (`EX_CONFIG` per sysexits(3))
  - Prints the following directive to stderr (showing the share path):
    ```
    Detected legacy .blk layout at <share-path>.
    v0.16+ requires CAS migration. Run:
        dfs migrate-to-cas --share <name>
    or, to migrate every share at once:
        dfs migrate-to-cas
    See docs/CONFIGURATION.md §migration.
    ```
  - Halts on the FIRST share that surfaces the legacy layout (per-share fail-fast).
  
  ### Running the migration
  
  The migration is an offline operation — stop the server first:
  ```bash
  dfs stop
  dfs migrate-to-cas
  ```
  
  Flags:
  
  | Flag | Default | Description |
  |------|---------|-------------|
  | `--storage-dir <path>` | (from config) | Storage root. Overrides config-derived value. |
  | `--share <name>` | (all shares) | Scope migration to one share. |
  | `--dry-run` | false | Walk the tree and report file count, total bytes, estimated dedup ratio, and ETA. Writes nothing. |
  | `--json` | false | Emit one JSON object per line of progress output for machine parsing. |
  | `--config <path>` | (default) | Override config file location. |
  
  Progress is reported to stdout ~once per second. With `--json`, each line is `{"ts":"<RFC3339>","share":"...","files_done":N,"bytes_done":N,"files_per_sec":f,"mib_per_sec":f,"dedup_hits":N,"eta_seconds":f}`.
  
  ### Crash safety
  
  The migration is idempotent. A per-share journal at `<storage_dir>/<share>/.dittofs-migrate-to-cas.state` records the last-completed file path and byte offset. If interrupted (Ctrl-C, kill -9, power loss), rerunning `dfs migrate-to-cas` resumes from the journaled position. The journal is deleted on full success.
  
  ### Verifying completion
  
  Success is recorded by a per-share sentinel file at `<share_dir>/.cas-migrated-v1` (one per share — `--share <name>` migrations produce just that share's sentinel; un-scoped migrations produce one sentinel per share at each share's completion, so partial-success states are operationally well-defined). Contents:
  ```json
  {
    "Version": "v1",
    "CompletedAt": "2026-05-20T14:30:00Z",
    "ToolVersion": "v1.0.0",
    "ShareDir": "/path/to/storage"
  }
  ```
  The sentinel is written via atomic rename only after every share completes — partial migrations cannot leave a sentinel behind. **Do not hand-create or hand-edit this file.** It is intended as a one-way irreversibility marker; modifying it bypasses the boot guard but cannot fix a half-migrated store.
  
  ### Recovery from a failed migration
  
  1. Inspect stderr for the file path + offset at which the migration halted.
  2. Inspect the journal at `<storage_dir>/<share>/.dittofs-migrate-to-cas.state` to confirm the resume point.
  3. Rerun `dfs migrate-to-cas` (optionally with `--share <name>` to scope to the affected share).
  4. If a chunk verification mismatch occurred (post-Put BLAKE3 disagreement), this indicates storage corruption between Put and re-Get — investigate the destination block store before retrying.
  
  ### See also
  
  - [docs/CLI.md — `dfs migrate-to-cas`](CLI.md#dfs-migrate-to-cas) for the full command-line reference.
