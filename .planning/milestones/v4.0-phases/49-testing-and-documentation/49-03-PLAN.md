  <action>
1. Create `test/e2e/multi_share_isolation_test.go` with `//go:build e2e` tag.

2. `TestMultiShareIsolation` main test function:
   - Start server, login as admin
   - Create 2 metadata stores (memory)
   - Create 2 local block stores (fs type, with DIFFERENT paths — use t.TempDir())
   - Optionally create 1-2 remote block stores
   - Create share A referencing meta1 + local1 (+ optional remote1)
   - Create share B referencing meta2 + local2 (+ optional remote2)
   - Mount both shares

3. Subtests (each as t.Run):

   a. **Data isolation** (both-NFS scenario):
      - Mount share A and share B via NFS
      - Create file "testfile.txt" on share A with content "share-a-data"
      - Verify share B does NOT contain "testfile.txt"
      - Create file "testfile.txt" on share B with content "share-b-data"
      - Verify share A's "testfile.txt" still contains "share-a-data"

   b. **Deletion isolation**:
      - Create files on both shares
      - Delete share A (via CLI `dfsctl share delete`)
      - Verify share B data is still intact and accessible

   c. **Concurrent writes**:
      - Use goroutines to write 10 files to share A and 10 files to share B simultaneously
      - Verify all 20 files written correctly with no corruption or cross-contamination

   d. **Cache independence**:
      - Configure share A with tiny cache (e.g., small write buffer) + remote store
      - Configure share B with larger cache + remote store
      - Write enough data to share A to trigger eviction
      - Verify share B cache is unaffected (use `dfsctl cache stats --share` to check)
      - This tests that per-share BlockStore instances have independent caches

   e. **Cross-protocol lock visibility** (NFS + SMB mixed mount):
      - Mount share A via SMB, mount share A via NFS
      - Mount share B via SMB
      - Lock file on share A via SMB
      - Verify NFS can see the lock on share A (same file)
      - Verify same filename on share B is NOT locked
      - This validates locks are per-share, not global

4. Also include a subtest with **same remote store** scenario:
   - Create 2 shares pointing to the SAME remote block store (same S3 bucket)
   - Write different files to each share
   - Verify payloadID namespacing keeps data separate (files from share A don't appear in share B)

5. Mount scenarios: Use both NFS-only and NFS+SMB mixed mounting as subtests.

6. Skip S3/Localstack-dependent subtests if Localstack unavailable.
